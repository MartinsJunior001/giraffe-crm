## Comandos

Pré-requisitos: Node 24 (`.nvmrc`) e `corepack enable` (pnpm 10 fixado em `packageManager`).

```bash
pnpm install                 # dev; em CI/validação: pnpm install --frozen-lockfile
pnpm dev                     # web (3000) + api (3001) em paralelo
pnpm lint / pnpm lint:fix    # ESLint 9 flat
pnpm format:check            # Prettier 3
pnpm typecheck               # tsc --noEmit em cada app
pnpm test                    # Vitest 4 em cada app — EXIGE PostgreSQL no ar (ver abaixo)
pnpm build                   # build de api e web
pnpm compose:up / :down      # execução conteinerizada
pnpm smoke                   # valida /health, /ready, /healthz e a casca de um ambiente JÁ no ar
```

Banco (a partir da Story 1.2 — `pnpm test` não passa sem ele):

```bash
cp .env.example .env                       # o Compose EXIGE as senhas; sem elas, `up` falha
docker compose up -d db                    # PostgreSQL 16 (host 127.0.0.1:5434)
pnpm --filter @giraffe/api db:migrate      # migrations (papel giraffe_migrator)
pnpm --filter @giraffe/api db:seed         # fixture: Orgs A, B e C
pnpm --filter @giraffe/api db:status       # estado das migrations
pnpm --filter @giraffe/api db:rollback     # ⚠️ DESTRUTIVO — reverte a migration mais recente
```

A suíte de RLS roda contra um PostgreSQL **real** e fica **vermelha** — não pulada — se o banco estiver fora: quem nega o acesso é o banco, e um mock não provaria nada. Banco indisponível é falha, não ausência de evidência.

Testes de um app ou arquivo (Vitest roda a partir da raiz de cada app):

```bash
pnpm --filter @giraffe/api test                       # só a API
pnpm --filter @giraffe/api exec vitest run test/env.test.ts
pnpm --filter @giraffe/api exec vitest run -t "nome do teste"
```

`pnpm smoke` **não sobe serviços** — exige `pnpm dev` ou `pnpm compose:up` antes. Aceita `API_URL`, `WEB_URL`, `SMOKE_TIMEOUT_MS`.

## Processo obrigatório (precede qualquer código)

Este repositório é governado por `.specify/memory/constitution.md`, cuja versão vigente e princípios estão registrados no próprio arquivo. Ela tem precedência sobre conveniência e hábito. Pontos que mudam o que você pode fazer:

- **Sequência oficial:** Documentação Base → BMAD → Spec Kit → Implementação → Validações por skills → Deploy. Não se escreve código de aplicação antes de a Story ter passado por BMAD e pelo Spec Kit (`specify → clarify → plan → checklist → tasks → analyze`; `converge` quando já há implementação parcial).
- **Gate pré-código:** `skills/pre-implementation-check.md` é obrigatório antes de implementar, corrigir bug funcional, criar migration, adicionar dependência ou mudar arquitetura. Ele produz um relatório com status `APROVADO` / `APROVADO COM RESSALVAS` / `BLOQUEADO`. Depois vem `safe-implementation`, e antes de concluir a Story: `security-check`, `observability-check` e (quando aplicável) `lgpd-check`, `migration-check`, `backup-check`, `performance-check` — todos em `skills/`.
- **Verificação documental antes de escrever código — obrigatória:** sempre que a implementação envolver biblioteca, framework, SDK, API, CLI ou serviço da stack, execute o `context7-check` antes de codificar, mesmo que a API pareça conhecida. Use como baseline a versão efetivamente declarada ou instalada no projeto, conforme `package.json`, `packageManager`, lockfile, Dockerfile ou configuração equivalente. Consulte preferencialmente o MCP do Context7 (`resolve-library-id` → `query-docs`). Quando o Context7 estiver indisponível ou não cobrir a tecnologia, consulte a documentação oficial atual e registre a fonte utilizada. Não invente assinaturas, opções de configuração ou versões. Quando a documentação atual contradisser o plano ou a arquitetura, registre a divergência e escale antes de implementar.
- **Sem antecipar escopo:** nada de Fase 2, nada de abstração especulativa (módulo vazio, repositório genérico, event bus) sem consumidor concreto. Proibição registrada explicitamente em `apps/api/src/kernel/README.md`.
- **Artefatos autoritativos não são editados diretamente pela implementação:** PRD, UX, Architecture Spine, `epics.md` e readiness report só mudam por seus workflows oficiais. O `sprint-status.yaml` e o status da Story só podem ser atualizados pelo workflow BMAD responsável, nunca por edição manual ou por uma implementação fora desse fluxo.
- **Commit:** não commitar antes do `commit-check`; nunca fazer push/deploy sem autorização explícita. Ver **Commit automático por seção**, abaixo.

Referências de decisão: `AD-*` (decisões de arquitetura) e invariantes vivem em `_bmad-output/planning-artifacts/architecture/*/ARCHITECTURE-SPINE.md`; a documentação de produto da Fase 1 está em `docs/01-documentacao-base/` (índice: `00-indice-fase-1.md`). `docs/_arquivo-legado/` **não é fonte oficial**; o protótipo HTML em `08-referencias-visuais/prototypes/` é referência visual, **não** modelo de dados nem arquitetura final.

## Commit automático por seção

Ao concluir cada seção, etapa ou Story com alterações versionáveis:

1. confirme que o escopo foi concluído e os gates obrigatórios estão verdes;
2. execute automaticamente a skill `commit-check`;
3. se o resultado for aprovado, execute automaticamente a skill `commit`;
4. crie um commit pequeno, atômico e com mensagem em português no padrão do projeto;
5. não inclua arquivos fora do escopo, temporários, segredos ou configurações locais;
6. não faça commit de trabalho parcial, bloqueado ou com testes vermelhos;
7. nunca execute `push`, merge, deploy ou mudança de branch sem autorização explícita.

Se a seção não gerar uma entrega versionável, não crie commit e registre apenas:

`SEM COMMIT — nenhuma alteração versionável nesta seção.`

## Git e integração

- **Branch por Story** (`story/<n>-<slug>`) ou por tarefa técnica (`tech/<slug>`). Nunca se trabalha direto em `main`.
- **Integração por Pull Request**, com o CI verde. O PR é o ponto em que a verificação deixa de ser local.
- **Merge commit (`--no-ff`), não squash.** Os commits desta base são atômicos e cada um carrega o _porquê_ no corpo — o commit que corrige um vazamento cross-tenant explica como ele foi reproduzido. Squash funde tudo numa mensagem só e joga fora exatamente a informação que um `git bisect` ou uma investigação de incidente vai procurar. Rebase reescreveria história já publicada.
- **Nunca `--force`** em história compartilhada, e nunca `--no-verify`.

## Invariantes conceituais (nunca erodir)

`Pipe ≠ Database` · `Card ≠ Registro` · `Fase ≠ Status do Card` · `Super Admin (Plataforma) ≠ Admin da Organização` · `Usuário ≠ Organização`. Identidade é Account global + Membership por Organização. Autorização é **deny-by-default** e `PERMISSÃO = AÇÃO + ESCOPO`; isolamento por Organização é o invariante-mãe. Os três Formulários (inicial, de fase, de database) são independentes.

## Arquitetura

Monorepo pnpm workspaces (`apps/*`), TypeScript estrito compartilhado via `tsconfig.base.json` (`strict` + `noUncheckedIndexedAccess`). O frontend consome apenas a API interna e nenhuma superfície de API pública é implementada. Nenhuma regra de domínio deve existir no frontend.

**Estado transitório da implementação — Story 2.11 (ciclo de vida do Card).** O L1 está fechado: existe banco com isolamento por RLS, autenticação e sessão (1.4/1.5), contexto de Organização resolvido no servidor (1.3), autorização por papel (1.6, CASL deny-by-default) e a casca navegável (1.7/1.8). O Épico 2 avançou: **existe `Pipe`** (2.1 — ciclo de vida/catálogo, Admin da Org), **`PipeGrant`** (2.2 — concessão de papel por Pipe + acesso por concessão), **`Phase`** (2.3 — Fase do fluxo: criar/renomear/reordenar/arquivar/restaurar, ordenação fracionária, invariante "≥1 Fase ativa") e o **domínio Formulário** (2.4): **catálogo canônico de tipos de Campo** (enum `FieldType`, 12 tipos, de plataforma), **`Form`** (Formulário de um contexto — `PIPE_INITIAL`/`PHASE`; `DATABASE` é contrato do E3, sem owner ainda) e **`Field`** (Campo — rótulo/tipo/ajuda/`typeConfig` JSON/valor padrão/posição fracionária/estado). Montagem = obter (getOrCreate lazy no adicionar; ler não cria), adicionar e reordenar Campo; contextos inicial e de Fase **independentes** (INV-FORM-01). Opções de Seleção vivem no `typeConfig` (JSON com UUID estável), não em tabela (DBT-2.4-OPCOES-JSON). O tipo Arquivo é gated (AD-28, fail-closed): `FILE_UPLOAD_ENABLED` default falso + função pura `podePublicarComArquivo` (contrato consumido pela publicação, 2.6 — DBT-2.4-FILE-GATE-CONSUMO). A **evolução segura de Campos** (2.5) fecha o L2 de configuração: editar rótulo/ajuda/valor padrão (não `type`, imutável), arquivar/restaurar Campo (transição reversível; restaura ao **final** da ordem ativa) e o **ciclo de opções de Seleção** (adicionar/renomear/reordenar/arquivar/remover). As opções seguem em JSON no `typeConfig` (Opção A — **sem migration**; `state`/`archivedAt` já existiam em `Field` desde a 2.4). `option-config.ts` é o núcleo puro dos invariantes das opções (id estável/único, `label` não confiável, allowlist de chaves, fail-closed, limites); o serviço reescreve o `typeConfig` inteiro num único `update` com **guarda otimista** (`typeConfig: { equals: <lido> }` → **409** em conflito, sem lost update silencioso). Sem exclusão: remover opção é UPDATE, nunca DELETE de linha. A **publicação** (2.6) fecha o ciclo do Formulário: publicar congela o rascunho **validado** numa **`FormVersion`** imutável (snapshot JSON integral dos Campos ativos, numerada por `@@unique([orgId, formId, version])`); despublicar zera o ponteiro `Form.publishedVersion` (preserva o histórico); ler devolve estado e snapshot. **Imutabilidade é do banco**: o runtime tem GRANT só `SELECT`/`INSERT` em `FormVersion` (sem UPDATE/DELETE). Validação de publicabilidade em `snapshot.ts` (fail-closed): sem Campo ativo, Seleção sem opção ativa, gate de Arquivo (`podePublicarComArquivo`) e `typeConfig` malformado → 400; o snapshot **não inventa obrigatoriedade** (não existe em `Field`). A publicação toca duas escritas (INSERT versão + UPDATE ponteiro) numa **transação interativa com contexto no client raiz** — o `withTenantContext` recusa `$transaction` no client estendido, mas o raiz roda o mesmo primitivo `set_config(..., true)`, agora com **fonte única** (`definirContextoOrg` em `tenant-context.ts`); concorrência de número resolve por `UNIQUE` → rollback integral → 409 (P2002/P2028). A **submissão interna** (2.7) abre o ciclo do Card: submeter o Formulário inicial **publicado** de um Pipe cria um **`Card`** na 1ª Fase ativa (menor `position`), que referencia a `FormVersion` publicada no ato (definição **congelada** — AD-12) e guarda os `valores` validados contra o snapshot em **JSONB por `Field.id`** (opção por `id`, nunca rótulo; sem tabela de valores por Campo — AD-11). Um evento **`CardHistory`** `CREATED` é escrito na **mesma transação** interativa (AD-13; mesmo primitivo `definirContextoOrg` da 2.6): não há Card sem evento. Validação em `submission.ts` (fail-closed): allowlist anti-mass-assignment (chave desconhecida → 400), tipo por Campo, Seleção por `id`; valor ausente é permitido (não há obrigatoriedade em `Field`). **Idempotência** por `@@unique([orgId, formId, idempotencyKey])` — retry devolve o Card existente; o conflito reconhece **P2002 e P2028** → idempotente ou 409, **nunca 500**. **GRANT como fronteira**: `Card` tem só `SELECT`/`INSERT` (a 2.7 apenas cria; UPDATE/DELETE ficam para a movimentação/ciclo de vida do Card — 2.14/2.11 — com consumidor e teste que provam o escopo) e `CardHistory` só `SELECT`/`INSERT` (append-only imutável, como `FormVersion`). A **submissão pública controlada** (2.8) abre o canal externo do Formulário inicial: um endpoint **não autenticado** (`POST /public/forms/:publicId/submit`) resolve o tenant por um **`PublicFormRoute` GLOBAL** (sem RLS por definição — AD-10, análogo a `Account`) que mapeia um `publicId` **opaco** para `(orgId, formId)`, entra em `withTenantContext(orgId)` e **RELÊ o Form sob RLS** — nunca confia em `orgId`/`formId`/`pipeId`/`formVersionId` do cliente; link inválido/revogado/não publicado/cross-tenant → **404 uniforme**; a resposta é só `{ ok: true }` (sem vazamento). A submissão vira uma **`SubmissaoPublica`** org-scoped com **ciclo próprio** (`PENDING`/`CONVERTED`/`REJECTED`) — **`Fase ≠ Status do Card`** preservado. **TRIAGE** (padrão) fica PENDING até a triagem; **DIRECT** converte na hora. Na triagem, **aprovar = converter é atômico** (AD-11): cria **1 Card** (`origin=PUBLIC`) na 1ª Fase ativa + evento `CREATED` na mesma transação raiz; **rejeitar** preserva (`REJECTED`, sem Card). A conversão trata **P2002 e P2028** → idempotente/409, **nunca 500**, com auditoria manual (FR-214). A autorização de revisão é por **capacidade explícita `reviewPublicSubmissions`** na `PipeGrant` (deny-by-default; Admin da Org implícito; demais só por concessão), reusando a guarda fina de `pipe-authz.ts` (`exigirRevisarSubmissoesPublicas`, C3 congelado). Baseline antiabuso: **rate limit atômico** por IP confiável + `publicId` (fail-closed → 429), **Arquivo bloqueado no público** (AD-28), sem CAPTCHA. **GRANT como fronteira**: `SubmissaoPublica` tem `SELECT`/`INSERT`/`UPDATE` — **sem DELETE** (preserva o dado do titular — LGPD; rejeitar é `state`); `PublicFormRoute` idem **sem DELETE** (revogar é `active=false`; rotacionar = revogar + criar novo `publicId`). O **Kanban e o espaço operacional do Card** (2.9) abrem a superfície de **LEITURA** sobre `Card`/`Phase` já materializados — **sem migration e sem GRANT novo** (o runtime segue sem UPDATE em `Card`; **mover é a 2.14**). `KanbanReadService` reusa `resolverPoderNoPipe` (basta **qualquer** poder no Pipe — inclusive **VIEWER concedido**, pois ler ≠ operar; sem acesso → 404 não-enumerante) e devolve as **capacidades** derivadas do poder do próprio principal (nunca revela ação alheia). `verKanban` lista as colunas (Fases ativas por `position`) com a contagem via **um `groupBy`** (sem N+1); a coluna **pagina por cursor determinístico** `[createdAt, id]` (teto 100 — NFR-3/4). `valores` (possível PII) só no **detalhe** do Card, nunca na lista nem em log; `orgId` fora da fronteira. A divergência "2.9 = movimentação" (comentário legado da migration da 2.7) foi resolvida a favor do `epics.md` (movimentação = 2.14); o design de movimentação vive no Apêndice A da spec da 2.9. O **acesso, Responsável e concessões de Card** (2.10) fecha o **DBT-2.2-ROLE-DORMENTE**: introduz a autorização **no nível do Card** sobre o substrato de Pipe, com três dados novos org-scoped. **`CardGrant`** (D-OA1) é a concessão DIRETA de acesso a UM Card (Observador = ler; operacional = operar [+`podeMover` — só o DADO da capacidade; a operação é a 2.14]); RLS+FORCE+WITH CHECK, GRANT `SELECT/INSERT/UPDATE` **sem DELETE** (revogar = `state`). **`CardResponsavel`** (D-OA2) é a atribuição de Responsável por Card, chave `membershipId`; mantém `Card` **append-only** — a 2.10 **não** abre GRANT de UPDATE em `Card` (o 1º UPDATE segue reservado à movimentação, 2.14; invariante preservado); `creator` continua sendo o `actorId` do `CREATED` (2.7), sem coluna nova. **`PipeGrant.restritoAoProprio`** é um modificador do Membro (novo DADO de autorização, análogo a `reviewPublicSubmissions` da 2.8; **não** é papel novo), exposto pelas rotas de grant. A resolução `resolverAcessoNoCard` (em `pipe-authz`) **compõe** papel-de-Pipe + concessão direta + "restrito ao próprio" + Responsável-atual: deny-by-default, sem acesso → **404 não-enumerante**, ler-sem-operar → **403**; **creator e histórico NUNCA concedem acesso** (SC-2105). Atribuir/remover Responsável exige **operar o Card** (`exigirOperarCard`, mais restrito que operar o Pipe — um Membro "restrito ao próprio" não mexe em Cards que não acessa) e o **alvo** precisa de acesso operacional prévio (SC-2101; atribuir não amplia acesso — SC-2102); conceder/revogar/listar exige **gerenciar o Pipe** (`exigirGerenciarPipe`). Cada mutação escreve um evento `CardHistory` (`RESPONSAVEL_ASSIGNED`/`CHANGED`/`REMOVED`, `ACCESS_GRANTED`/`ACCESS_REVOKED`) na **mesma transação** interativa no client raiz (`definirContextoOrg`, auditoria manual FR-214). Idempotência por índice PARCIAL de unicidade ativa (1 Responsável ativo por Card; 1 concessão ativa por (Card, pessoa)); conflito reconhece **P2002 e P2028** → 409, nunca 500. O **contrato de Membership** (D-OA3) é materializado como **funções PURAS** (`membership-contract.ts`: preflight de encerramento + handler pós-alteração), **sem chamador** — o E8 as consumirá; o preflight nasce **vacuamente verdadeiro** (a regra "Card exige Responsável ativo" não existe para Card na Fase 1 — DIV-3, não inventada); reativar não restaura acesso; `creator` preservado por construção. O **ciclo de vida do Card** (2.11) introduz o eixo de estado — `ATIVO`/`FINALIZADO`/`ARQUIVADO` (`reaberto`/`restaurado` são **transições**, não estados) — independente da Fase (`Fase ≠ Status do Card`) e da saúde temporal (2.13). Finalizar/reabrir/arquivar/restaurar são transições **atômicas**, **idempotentes** e **auditadas** (evento próprio no `CardHistory`: `FINALIZED`/`REOPENED`/`ARCHIVED`/`RESTORED`, na mesma transação interativa — `definirContextoOrg`, auditoria manual FR-214). O **estado anterior ao arquivamento** vive em `Card.previousLifecycleState` e a restauração o devolve (FINALIZADO→arquivar→restaurar volta a FINALIZADO, não a ATIVO). A decisão de QUAL transição é o núcleo **puro** `planejarTransicao`; o serviço a aplica com **guarda otimista** (`updateMany where lifecycleState = <lido>` → reconsulta → idempotente/**409**; P2002/P2028 → 409, nunca 500); transição inválida a partir do estado atual → **409**. Autorização: **operar o Card** (`exigirOperarCard`, 2.10) — 404 sem acesso, 403 só-lê. **O ponto arquitetural:** o ciclo de vida é o **1º UPDATE de `Card`** em runtime, e ele é **column-scoped** — `GRANT UPDATE ("lifecycleState", "previousLifecycleState", "updatedAt") ON "Card"`, e **só** isso; `phaseId` (movimentação, 2.14), `valores` e `orgId` seguem **sem** GRANT de UPDATE (tentativa bate em `permission denied` — provado no `card-lifecycle-rls`), reconciliando "evoluir estado sim, mover/reescrever não" exatamente como a migration da 2.7 antecipou. O detalhe do Card (2.9) passa a expor `lifecycleState` (a 2.13 consumirá); a **re-filtragem da LISTA** do Kanban por estado é da 2.13, não desta Story.

**O papel de Pipe deixou de ser dormente para _config_.** Na 2.2 o `PipeRole` era armazenado mas inerte (decisão SC-222=B). Na 2.3, **gerenciar Fases é "config do Pipe"** (PRD §7) e ativou o diferencial; na 2.4 **configurar Formulário (inicial e de Fase) também é config do Pipe** (D3.2) e na 2.5 **evoluir Campos (editar/arquivar/opções)** e na 2.6 **publicar/despublicar Formulário idem** — todas **reusam a mesma resolução**: gerencia o **Admin da Org** (qualquer Pipe) **ou** o **Admin do Pipe** (concessão `role=ADMIN` ACTIVE, com `Membership` ACTIVE); MEMBER/VIEWER concedidos só leem. Essa resolução foi **extraída** de `PhasesService` para o helper compartilhado `pipes/pipe-authz.ts` (`resolverPoderNoPipe`/`exigirGerenciarPipe`), consumido por Fases, Formulários, evolução de Campos e publicação — uma única guarda fina (DBT-AUTHZ-01), sem tocar o guard/`ability.ts` (C3 congelado). O Formulário de Fase e seus Campos resolvem o poder pelo Pipe dono da Fase (`phase.pipeId`). Os localizadores de Formulário foram extraídos para `forms/form-locate.ts`, compartilhados por `FormsService` (montagem, 2.4), `FieldsService` (evolução, 2.5) e `FormPublicationService` (publicação, 2.6). Na **2.7 o papel deixou de ser dormente para _operação_**: submeter o Formulário inicial exige **operar** o Pipe (`exigirOperarPipe`, `Poder = gerenciar > operar > ler`) — Admin da Org e **Membro do Pipe** submetem (poder do Membro **ativado**), Viewer concedido só lê (403), sem acesso 404. Na **2.8** a `PipeGrant` ganhou uma **capacidade** (não um papel): `reviewPublicSubmissions` habilita **revisar submissões públicas** (`exigirRevisarSubmissoesPublicas`), deny-by-default e concedida explicitamente — Admin da Org implícito; a config do acesso público (habilitar/revogar/rotacionar) reusa `exigirGerenciarPipe`. Na **2.10 o DBT-2.2-ROLE-DORMENTE foi fechado**: o acesso passou a ser resolvido também **no nível do Card** (`resolverAcessoNoCard`/`exigirOperarCard`/`exigirLerCard` em `pipe-authz`), compondo o papel-de-Pipe com a **concessão direta** (`CardGrant`), o modificador **`restritoAoProprio`** do Membro e o **Responsável-atual** (`CardResponsavel`) — sempre no serviço, sem tocar o guard/`ability.ts` (C3 congelado).

**Não** existe ainda: **mudança de tipo** de Campo e as **travas de arquivamento sob uso** (dependem de submissões correntes, 2.11+), a **movimentação do Card entre Fases** (2.14 — por isso `Card` só tem GRANT de UPDATE **column-scoped** para o estado, jamais `phaseId`), a **saúde temporal do Card** e sua precedência de apresentação (`ok`/`atrasado`/`vencido`/`expirado` — 2.13; e a re-filtragem da LISTA do Kanban pelo estado de ciclo de vida), os **marcos por Fase** e a referência temporal de entrada (2.12), a **operação de "Mover Card"** (a 2.10 guarda só o DADO `podeMover`; a operação é a 2.14), o **consumo do contrato de Membership** (as funções puras existem, mas o ciclo E8 que as chama não), a **re-filtragem da LEITURA do Kanban** pelo `restritoAoProprio`/concessão direta (a 2.10 escopa a autorização de OPERAÇÃO no Card; a listagem 2.9 ainda autoriza por poder de Pipe — sem consumidor concreto para o filtro fino de leitura), a **leitura do `CardHistory`** no painel (2.17; a 2.9 só estrutura o painel e a 2.10 amplia a taxonomia com os eventos de Responsável/concessão), a **pré-visualização** com submissão simulada, nem Databases (o contexto Database é só contrato, E3). As travas "não arquivar Pipe com Cards ativos" (2.11) e "não arquivar Fase com Cards ativos" seguem **contrato futuro** — já existe tabela `Card`, mas nenhuma trava foi materializada antes do seu consumidor (AD-11).

- `apps/api` — NestJS 11. `main.ts` faz **fail-fast**: `getEnv()` valida o ambiente (Zod, `kernel/config/env.ts`) _antes_ de o Nest subir; CORS restrito sem wildcard; `enableShutdownHooks()`. `AppModule` configura Pino via `forRootAsync` (a validação de env fica no `useFactory`, para que importar o módulo em testes não tenha efeito colateral), com redaction de `authorization`/`cookie`/`set-cookie` e supressão de log dos probes.
- `apps/api/src/kernel/` — fronteira técnica transversal mínima (AD-4/AD-5). **Regra de negócio nunca vive aqui.** Hoje `config/`, `db/`, `context/` (contexto de Organização por requisição) e `authz/` (substrato CASL + guard deny-by-default).
- `apps/api/src/pipes/` — domínio do Épico 2. `PipesService`/`PipeGrantsService`/`PhasesService`/`FormsService` fazem **toda** query por `withTenantContext` — não há um único `where orgId` manual, porque um `where` se esquece e a policy não. Nenhuma rota aceita `orgId` do cliente. **Não há rota de exclusão** em nenhuma entidade: o runtime **não tem GRANT de DELETE** em `Pipe`/`PipeGrant`/`Phase`/`Form`/`Field`. Arquivar/revogar é mudança de estado (preserva os dados) e idempotente; transições respondem **200**, criação **201**. Subdomínios: `grants/` (concessões de papel por Pipe, 2.2), `phases/` (Fases, 2.3), `forms/` (Formulário/Campos, 2.4), `cards/` (submissão interna/Card, 2.7 + **leitura do Kanban/detalhe**, 2.9 — `kanban-read.service`/`kanban.controller` — + **acesso/Responsável/concessões de Card**, 2.10 — `cards/access/`: `card-access.service`/`card-access.controller` e a função-contrato pura `membership-contract.ts` — + **ciclo de vida do Card**, 2.11 — `cards/lifecycle/`: `card-lifecycle.service`/`card-lifecycle.controller` e o núcleo puro `card-lifecycle.transitions.ts`) e `public-submissions/` (canal público + triagem + config, 2.8). O runtime **não tem GRANT de DELETE** em nenhuma entidade do domínio (`Pipe`/`PipeGrant`/`Phase`/`Form`/`Field`/`Card`/`CardHistory`/`SubmissaoPublica`/`CardGrant`/`CardResponsavel`): revogar/remover/arquivar é `state`. `Card` ganhou seu 1º GRANT de **UPDATE column-scoped** na 2.11 (só `lifecycleState`/`previousLifecycleState`/`updatedAt`; **não** `phaseId`/`valores`). A autorização FINA por recurso ("config do Pipe" — quem gerencia Fases e monta Formulários; quem revisa submissões públicas; quem opera/concede acesso a um Card) vive **no serviço** via o helper compartilhado `pipe-authz.ts` (DBT-AUTHZ-01), não no guard. `SubmissaoPublica` é org-scoped (RLS); `PublicFormRoute` é a única tabela **global** do domínio (resolução de tenant pré-contexto — AD-10). Nenhuma tem GRANT de DELETE.
- `apps/api/src/kernel/db/` — `PrismaService` (client do runtime, papel `giraffe_app`) e `tenant-context.ts`. **A conexão é preguiçosa por decisão**: um `$connect()` ansioso mataria o processo com o banco fora no boot, antes de abrir a porta — sem `/health`, sem `/ready`, sem 503.
- `apps/web` — Next.js 16 (App Router, `output: standalone`), React 19, Tailwind 4. `lib/env.ts` lê `API_BASE_URL` (variável de **servidor**, deliberadamente sem `NEXT_PUBLIC_`); `lib/api.ts` consulta `/health` com timeout e devolve **estado honesto e sanitizado** — falha nunca vaza URL interna, stack ou segredo. `GET /healthz` é a liveness da Web: rota local, sem I/O, **não** consulta a API (a saúde do container não pode depender de um serviço terceiro).

**Health vs. readiness são semanticamente distintos**: `GET /health` (liveness, nunca toca o banco) e `GET /ready` (readiness). Desde a Story 1.2, `/ready` **consulta o banco** e devolve **503** quando ele não está apto — a sonda lê uma tabela do schema (`LIMIT 0`), o que prova conexão, migrations aplicadas e GRANT concedido. Um `SELECT 1` provaria só o socket, e um container com o schema ausente responderia `200 ok` para falhar em toda requisição real. Nenhum payload expõe versão, variáveis, paths ou segredos.

### Isolamento multi-tenant (o invariante-mãe)

- Quem isola é o **banco**, não a aplicação: `Organization`, `Membership`, `Pipe`, `PipeGrant`, `Phase`, `Form`, `Field`, `FormVersion`, `Card`, `CardHistory`, `SubmissaoPublica`, `CardGrant` e `CardResponsavel` têm `ENABLE` **e** `FORCE ROW LEVEL SECURITY`. `Account` (AD-10) e `PublicFormRoute` (2.8, resolução de tenant pré-contexto) são **globais e sem RLS** — nenhum pertence a um tenant.
- **Toda tabela organizacional nova replica esse padrão** (`Pipe`, `PipeGrant`, `Phase`, `Form`, `Field`, `Card`, `CardHistory`, `SubmissaoPublica`, `CardGrant`, `CardResponsavel` no Épico 2): policies `select/insert/update/delete` por `orgId = current_org_id()`, com `WITH CHECK` no INSERT **e** no UPDATE — sem o `WITH CHECK`, um INSERT com `orgId` alheio seria aceito e ficaria invisível, e um UPDATE poderia **mover** a linha para outra Organização.
- **Nenhuma query organizacional pode passar fora de `withTenantContext()`/`withAccountContext()`** (`kernel/db/tenant-context.ts`). O contexto é definido por **transação** (`set_config(..., true)`); com `false` ele grudaria na conexão e vazaria pelo pool.
- **Não existe, e não pode passar a existir, caminho de bypass de RLS alcançável em runtime** (AD-6). O exemplo oficial do Prisma sugere uma `bypass_rls_policy` — proibida aqui.
- **Dois papéis de banco, nunca o mesmo:** `giraffe_app` (runtime; sem `BYPASSRLS`, não é dono das tabelas, DML mínima) e `giraffe_migrator` (dono do schema, só migrations). O processo que atende requisição **nunca** tem em mãos a credencial do dono.
- **O `GRANT` é fronteira de segurança, não detalhe administrativo.** Onde a RLS não alcança, é ele que nega: `Account` não tem policy, então o runtime tem **só `SELECT`** nela — com `DELETE`, a cascata da FK apagaria Memberships de **todas** as Organizações, porque ações referenciais rodam com bypass de row security. Pela mesma razão o runtime não pode criar nem apagar `Organization`. Em `Pipe`, o runtime tem `SELECT/INSERT/UPDATE` e **não** `DELETE`: "sem exclusão definitiva" é garantido pelo banco, não pela ausência de rota — uma rota de DELETE acrescentada por engano amanhã bateria em `permission denied`. Ao conceder um privilégio novo, escreva o teste que prova o escopo dele.
- `$transaction` no client com contexto é **recusada** (erro de compilação e de runtime): a extensão fecha sobre o client raiz, então uma transação externa rodaria em outra conexão, sem contexto. Transação com contexto é escopo da Story 1.3.
- **Mutação de entidade organizacional entra na trilha de auditoria** (`MODELOS_AUDITADOS` em `tenant-context.ts` — hoje `Organization`, `Membership`, `Pipe`, `PipeGrant`, `Phase`, `Form`, `Field`, `FormVersion`, `Card`, `CardHistory`, `SubmissaoPublica`, `PublicFormRoute`, `CardGrant`, `CardResponsavel`), inclusive a **tentativa negada**: um `updateMany` filtrado pela policy volta `{ count: 0 }` com sucesso aparente, e sem isso a tentativa mais óbvia de vandalismo cross-tenant seria registrada como `allowed`. O troco é um falso positivo conhecido: uma operação idempotente legítima (arquivar um Pipe/Fase já arquivado) também produziria `count: 0` — por isso os caminhos idempotentes retornam SEM emitir o `updateMany`.

## Convenções que o código já assume

- **Segredos e configuração:** `.env` nunca é versionado (só `.env.example`); segredos vêm do cofre/ambiente, nunca de imagem, log ou health. Variável obrigatória ausente → falha honesta, sanitizada, listando apenas _nomes_ de variáveis (ver `ConfigValidationError`).
- **Logs:** estruturados (Pino), sempre sanitizados — nunca senhas, tokens, cookies, headers de auth, corpos de e-mail, prompts/respostas de IA ou PII desnecessária.
- **Versões:** fixadas no `pnpm-lock.yaml`. Nunca `latest`; não trocar a stack sem decisão arquitetural registrada. A API de qualquer biblioteca se confere no MCP do Context7 (ver gate acima), não de memória.
- **Migrations:** versionadas e aplicadas como **etapa controlada** (`db:migrate`), nunca no boot do container — um container que migra ao subir transforma cada réplica e cada restart numa tentativa concorrente de DDL. `prisma db push` não substitui migration versionada. O bootstrap dos **papéis** (`apps/api/prisma/bootstrap/00-roles.sql`) é idempotente, roda com papel administrativo e **precede** as migrations: criar papel exige privilégio que nem o migrator tem.
- **Testes:** Vitest com `include: ['test/**/*.test.ts']` — testes ficam em `apps/*/test/`, fora de `src/`. O contrato HTTP (`/health`, `/ready`) e o isolamento por RLS são cobertos por **teste de integração real** (AppModule em porta efêmera; PostgreSQL de verdade): um teste que só chamasse a função de payload continuaria verde se a rota fosse renomeada, e um mock de banco não provaria isolamento nenhum. Os arquivos de teste rodam em **paralelo** — Orgs A e B são fixture de **leitura**; escreva na **Org C**, ou o seu teste quebra o do vizinho por contagem. O typecheck da API cobre `src` **e** `test` (`tsconfig.json`); o build usa `tsconfig.build.json`, que exclui os testes de `dist`.
- **Um teste pode passar pelo motivo errado.** Nesta base isso já aconteceu duas vezes: o `create` do Prisma emite `INSERT ... RETURNING`, e o RETURNING esbarra na policy de **SELECT** — o teste de inserção cruzada ficava verde mesmo com o `WITH CHECK` desligado (por isso existe a versão com `createMany`, que não tem RETURNING). E o teste de "não é dono das tabelas" nunca olhava para `relowner`. Ao escrever um teste de segurança, **prove a fase vermelha**: quebre a policy de propósito e confirme que ele falha.
- **ESLint ignora** `docs/`, `_bmad/`, `_bmad-output/`, `skills/`, `specs/`, `.specify/`, `.github/`, `.agent*/` — são tooling e documentação, não código de aplicação.
- **CI:** `.github/workflows/ci.yml` — 4 jobs (`qualidade`, `testes`, `containers`, `seguranca`). O banco do CI sobe pelo **Docker Compose**, para que exista **uma única** definição de provisionamento de papéis; reescrevê-la no YAML criaria uma segunda verdade, e a que vale em produção seria a que ninguém testa. Senhas de CI são geradas por execução; actions são fixadas por **SHA**, nunca por tag (tag é mutável).
- **Testes verdes ≠ afirmação:** um critério de aceite só é marcado como concluído com evidência de execução real (Constitution X).

# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

> Idioma do projeto: **português (pt-BR)**, incluindo comentários de código, documentação e mensagens de commit.

## Comandos

Pré-requisitos: Node 24 (`.nvmrc`) e `corepack enable` (pnpm 10 fixado em `packageManager`).

```bash
pnpm install                 # dev; em CI/validação: pnpm install --frozen-lockfile
pnpm dev                     # web (3000) + api (3001) em paralelo
pnpm lint / pnpm lint:fix    # ESLint 9 flat
pnpm format:check            # Prettier 3
pnpm typecheck               # tsc --noEmit em cada app
pnpm test                    # Vitest 4 em cada app — EXIGE PostgreSQL no ar (ver abaixo)
pnpm build                   # build de api e web
pnpm compose:up / :down      # execução conteinerizada
pnpm smoke                   # valida /health, /ready, /healthz e a casca de um ambiente JÁ no ar
```

Banco (a partir da Story 1.2 — `pnpm test` não passa sem ele):

```bash
cp .env.example .env                       # o Compose EXIGE as senhas; sem elas, `up` falha
docker compose up -d db                    # PostgreSQL 16 (host 127.0.0.1:5434)
pnpm --filter @giraffe/api db:migrate      # migrations (papel giraffe_migrator)
pnpm --filter @giraffe/api db:seed         # fixture: Orgs A, B e C
pnpm --filter @giraffe/api db:status       # estado das migrations
pnpm --filter @giraffe/api db:rollback     # ⚠️ DESTRUTIVO — reverte a migration mais recente
```

A suíte de RLS roda contra um PostgreSQL **real** e fica **vermelha** — não pulada — se o banco estiver fora: quem nega o acesso é o banco, e um mock não provaria nada. Banco indisponível é falha, não ausência de evidência.

Testes de um app ou arquivo (Vitest roda a partir da raiz de cada app):

```bash
pnpm --filter @giraffe/api test                       # só a API
pnpm --filter @giraffe/api exec vitest run test/env.test.ts
pnpm --filter @giraffe/api exec vitest run -t "nome do teste"
```

`pnpm smoke` **não sobe serviços** — exige `pnpm dev` ou `pnpm compose:up` antes. Aceita `API_URL`, `WEB_URL`, `SMOKE_TIMEOUT_MS`.

## Processo obrigatório (precede qualquer código)

Este repositório é governado por `.specify/memory/constitution.md`, cuja versão vigente e princípios estão registrados no próprio arquivo. Ela tem precedência sobre conveniência e hábito. Pontos que mudam o que você pode fazer:

- **Sequência oficial:** Documentação Base → BMAD → Spec Kit → Implementação → Validações por skills → Deploy. Não se escreve código de aplicação antes de a Story ter passado por BMAD e pelo Spec Kit (`specify → clarify → plan → checklist → tasks → analyze`; `converge` quando já há implementação parcial).
- **Gate pré-código:** `skills/pre-implementation-check.md` é obrigatório antes de implementar, corrigir bug funcional, criar migration, adicionar dependência ou mudar arquitetura. Ele produz um relatório com status `APROVADO` / `APROVADO COM RESSALVAS` / `BLOQUEADO`. Depois vem `safe-implementation`, e antes de concluir a Story: `security-check`, `observability-check` e (quando aplicável) `lgpd-check`, `migration-check`, `backup-check`, `performance-check` — todos em `skills/`.
- **Verificação documental antes de escrever código — obrigatória:** sempre que a implementação envolver biblioteca, framework, SDK, API, CLI ou serviço da stack, execute o `context7-check` antes de codificar, mesmo que a API pareça conhecida. Use como baseline a versão efetivamente declarada ou instalada no projeto, conforme `package.json`, `packageManager`, lockfile, Dockerfile ou configuração equivalente. Consulte preferencialmente o MCP do Context7 (`resolve-library-id` → `query-docs`). Quando o Context7 estiver indisponível ou não cobrir a tecnologia, consulte a documentação oficial atual e registre a fonte utilizada. Não invente assinaturas, opções de configuração ou versões. Quando a documentação atual contradisser o plano ou a arquitetura, registre a divergência e escale antes de implementar.
- **Sem antecipar escopo:** nada de Fase 2, nada de abstração especulativa (módulo vazio, repositório genérico, event bus) sem consumidor concreto. Proibição registrada explicitamente em `apps/api/src/kernel/README.md`.
- **Artefatos autoritativos não são editados diretamente pela implementação:** PRD, UX, Architecture Spine, `epics.md` e readiness report só mudam por seus workflows oficiais. O `sprint-status.yaml` e o status da Story só podem ser atualizados pelo workflow BMAD responsável, nunca por edição manual ou por uma implementação fora desse fluxo.
- **Commit:** não commitar antes do `commit-check`. Push, PR, merge e deploy em **staging** são pré-autorizados pelo **Protocolo Autônomo de Aceleração** (abaixo); **produção** exige a aprovação consolidada de go/no-go. Ver **Commit automático por seção**, abaixo.

Referências de decisão: `AD-*` (decisões de arquitetura) e invariantes vivem em `_bmad-output/planning-artifacts/architecture/*/ARCHITECTURE-SPINE.md`; a documentação de produto da Fase 1 está em `docs/01-documentacao-base/` (índice: `00-indice-fase-1.md`). `docs/_arquivo-legado/` **não é fonte oficial**; o protótipo HTML em `08-referencias-visuais/prototypes/` é referência visual, **não** modelo de dados nem arquitetura final.

## Commit automático por seção

Ao concluir cada seção, etapa ou Story com alterações versionáveis:

1. confirme que o escopo foi concluído e os gates obrigatórios estão verdes;
2. execute automaticamente a skill `commit-check`;
3. se o resultado for aprovado, execute automaticamente a skill `commit`;
4. crie um commit pequeno, atômico e com mensagem em português no padrão do projeto;
5. não inclua arquivos fora do escopo, temporários, segredos ou configurações locais;
6. não faça commit de trabalho parcial, bloqueado ou com testes vermelhos;
7. `push`, PR, merge de PR verde e deploy em **staging** seguem o **Protocolo Autônomo de Aceleração** (abaixo) e não exigem autorização caso a caso; **produção** exige a aprovação consolidada de go/no-go.

Se a seção não gerar uma entrega versionável, não crie commit e registre apenas:

`SEM COMMIT — nenhuma alteração versionável nesta seção.`

## Git e integração

- **Branch por Story** (`story/<n>-<slug>`) ou por tarefa técnica (`tech/<slug>`). Nunca se trabalha direto em `main`.
- **Integração por Pull Request**, com o CI verde. O PR é o ponto em que a verificação deixa de ser local.
- **Merge commit (`--no-ff`), não squash.** Os commits desta base são atômicos e cada um carrega o _porquê_ no corpo — o commit que corrige um vazamento cross-tenant explica como ele foi reproduzido. Squash funde tudo numa mensagem só e joga fora exatamente a informação que um `git bisect` ou uma investigação de incidente vai procurar. Rebase reescreveria história já publicada.
- **Nunca `--force`** em história compartilhada, e nunca `--no-verify`.

## Protocolo Autônomo de Aceleração

Obrigatório para todas as sessões e agentes. Ele define **papéis, ritmo, autonomia, comunicação e continuidade** — não redefine os gates: quais skills rodam e quando continua sendo o que diz **Processo obrigatório**, e o que nunca pode ser cortado continua sendo **Isolamento multi-tenant** e **Convenções que o código já assume**. A camada de **Épico** — ordenação, proporcionalidade dos gates ao risco e não-duplicação de validação — está em **Protocolo Autônomo Fast Track por Épicos**, abaixo.

### Objetivo operacional

Implementar o MVP com a maior velocidade **segura**, priorizando: caminho crítico P0 → código funcional → testes e correções → integração e CI → staging → segurança/LGPD/backup/observabilidade/rollback → documentação necessária. P1 só quando não prejudicar o P0. Planejamento e relatórios servem à implementação; não a substituem.

### Autonomia

Até staging, trabalhe de forma autônoma. Não peça autorização para ações rotineiras dentro do escopo aprovado: investigar código/banco/testes/logs/CI; criar e editar arquivos; implementar Stories; corrigir bugs; criar e executar testes; rodar lint, typecheck, build e smoke; criar commits atômicos; fazer push; abrir e atualizar PRs; corrigir revisões; repetir CI até verde; **fazer merge de PR verde e aprovado**; executar closure; atualizar status e checkpoints; fazer deploy e validações **em staging**; avançar para a próxima Story P0 elegível.

Checkpoints são **informativos**, não pedidos de autorização — depois de apresentar um, continue automaticamente. **Produção exige uma única aprovação consolidada de go/no-go.**

A autonomia é de **ritmo**, não de alçada: restrições de segurança e de permissão da ferramenta continuam valendo, e nada aqui autoriza ação destrutiva ou bypass de controle.

### Fonte de verdade

Reconcilie **sempre nesta ordem**, do mais autoritativo ao mais volátil:

1. `origin/main` — código efetivamente mergeado;
2. **PRs e CI do GitHub** — o estado verificado do trabalho em voo;
3. `sprint-status`;
4. artefatos BMAD e Spec Kit (PRD/UX/Arquitetura/Épicos/Stories);
5. branches e worktrees;
6. **board operacional**;
7. MEMORY e checkpoints.

PRs e CI vêm **logo depois do `main`** porque são a evidência mecânica mais fresca do que existe: um `sprint-status` desatualizado ou um checkpoint escrito antes de uma queda descrevem intenção, enquanto um check do CI descreve fato.

Evidência real prevalece sobre checkpoint antigo. **Um checkpoint nunca prevalece sobre o estado real do repositório**, e **nunca sobrescreva código mergeado com informação histórica desatualizada.** Isso não torna o checkpoint um pedido de autorização (ver **Autonomia**): ele acelera a retomada, mas quem decide o que é verdade é o repositório.

### Fluxo obrigatório

Cada Story percorre, proporcionalmente ao risco: BMAD/create-story (ou validação da Story) → Spec Kit (`specify` → `clarify` **só para ambiguidade material** → `plan` → `checklist` → `tasks` → `analyze`) → `pre-implementation-check` → implementação → testes → revisão adversarial → checks aplicáveis de security/LGPD/observability/migration/backup → correções → `commit-check` → PR → CI → merge → closure → atualização de status → checkpoint durável → próxima Story elegível.

Etapas podem ser **consolidadas** quando seguro, mas suas **evidências não podem ser eliminadas**. Não pare depois de produzir apenas plano, spec, auditoria ou relatório quando houver trabalho executável.

A atualização de `sprint-status.yaml` e do status da Story é automática **pelo workflow BMAD responsável** — a autonomia acelera o disparo do workflow, **não** autoriza edição manual desses artefatos (ver **Processo obrigatório**).

### Papéis

Papéis são **funções**, não pessoas nem quantidade fixa de sessões. Um papel pode estar vago; nenhum papel acumula merge com implementação da mesma Story.

- **Lane 0 — Orquestrador e Release.** Fonte central de ownership; escolhe a próxima Story e a atribui; mantém a fila de integração e a fila de migrations; acompanha PR, CI, staging e closure; mantém o board. **Não implementa código funcional de Story** e é o **único agente autorizado a ordenar merge e closure**. A separação existe para que quem integra não seja quem se autoavalia.
- **Writer (A, B, …) — implementação.** Writer **exclusivo de uma** Story, com branch e worktree exclusivos. Implementa, testa, corrige findings e prepara o PR. **Não faz closure** e **não se autoatribui Story**. Um Writer não edita arquivo, contrato, migration ou superfície reservada à Story de outro Writer.
- **QA/Revisor — verificação.** Revisão **read-only**: não escreve na branch do Writer. Publica os findings **no PR**. Verifica segurança, isolamento/RLS, migrations, rollback, testes negativos e regressão. **Não faz merge nem closure.**

### Ownership e worktrees

- Um único Writer funcional **por Story** e um único Writer **por worktree**; uma Story em andamento por Writer.
- **Ownership é atribuído pela Lane 0 e registrado antes da escrita.** Nenhum arquivo ou contrato crítico com Writers concorrentes; nenhuma migration concorrente — a fila de migrations é da Lane 0.
- Antes de editar, confira branches, worktrees, Writers e alterações existentes. Se outro Writer estiver ativo, **não duplique o trabalho** e preserve as alterações dele.
- **Não** use `force`, `reset`, `clean` nem remoção de worktree para resolver colisão — conflito se resolve **semanticamente**, sem apagar trabalho válido.
- Sessão sem ownership funcional assume uma lane **não-funcional** — Orquestração, revisão ou planejamento — e **nunca uma Story**: assumir Story sem atribuição da Lane 0 é exatamente a colisão que este protocolo previne.

### Paralelismo seguro

Paralelize somente Stories **realmente** independentes. Lane 0: orquestração, integração, `main`, CI, staging e release. Lane 1: caminho crítico. Lanes adicionais: fluxos P0 independentes. Lane de qualidade: E2E, segurança, LGPD, migrations, backup, observabilidade e rollback.

WIP máximo de uma Story por Writer; planejamento no máximo uma Story à frente; branches curtas; PRs pequenos e verticais; integração frequente; `main` sempre verde. Nenhum paralelismo artificial e nenhuma edição concorrente do mesmo contrato.

### Barramento de comunicação

Agentes se comunicam por **artefatos que persistem**, nunca por transcrição de chat. O dono **não é o transportador de mensagens rotineiras** entre sessões.

**Depois que existe PR, o PR é o canal oficial da Story.** Os findings do QA são publicados no PR; o Writer responde e corrige **no mesmo PR**; evidências de teste são anexadas ao PR ou aos arquivos de gates; a Lane 0 acompanha o PR e o CI diretamente, sem depender de alguém repetir o resultado. Decisão material fica registrada no PR **e** na documentação durável correspondente — um comentário de PR sozinho não é onde uma decisão de arquitetura sobrevive.

**Antes de existir PR**, ownership e estado resumido ficam no **board**. Só a Lane 0 altera o ownership oficial; Writers consultam o board e fazem anticolisão **antes** de escrever.

### Board operacional

`_bmad-output/implementation-artifacts/board-operacional.md` — colunas: Story · estado · Writer · QA · branch/worktree · PR · CI · bloqueio · próxima ação · prioridade.

**Escrito exclusivamente pela Lane 0.** Writers e QA leem, não editam: um board que todos escrevem vira o conflito de merge que ele deveria evitar. Ele é **fotografia operacional** — nunca fonte superior ao GitHub ou ao `origin/main` (ver **Fonte de verdade**). Divergiu do repositório, o board está errado.

É também onde vive o estado que **não pode entrar neste arquivo**: SHA atual, Story em andamento, PR atual, nomes de branch e worktree, datas e ownership do momento. Este arquivo descreve **processo estável**; estado volátil que virasse instrução permanente envelheceria em silêncio e passaria a mentir.

### Roteamento determinístico

Eventos nomeados, para que cada agente saiba se a próxima ação é dele sem precisar perguntar:

| Evento                           | Quem age        | Ação                                       |
| -------------------------------- | --------------- | ------------------------------------------ |
| `STORY_ASSIGNED`                 | Writer indicado | anticolisão e reconciliação, depois inicia |
| `IMPLEMENTATION_READY`           | Writer          | publica branch/PR com evidências           |
| `CI_GREEN_READY_FOR_QA`          | QA              | inicia a revisão                           |
| `QA_FINDINGS`                    | Writer da Story | corrige no próprio PR                      |
| `QA_APPROVED`                    | Lane 0          | valida a fila de integração                |
| `MERGED`                         | Lane 0          | executa o closure BMAD                     |
| `CLOSURE_DONE`                   | Lane 0          | libera a próxima Story elegível            |
| `BLOCKED` por mais de duas horas | Lane 0          | decide a alternativa                       |

**Escalar ao dono** apenas: decisão de produto; migration destrutiva; segredo indisponível; risco Critical/High sem mitigação — os mesmos casos de **Condições de parada**, aqui sem duplicar a lista.

### Limite da automação entre terminais

Este arquivo define **comportamento**, não dispara processo: um documento não acorda outra sessão. A consequência é operacional e precisa estar explícita:

- um terminal ocioso **não assume Story sozinho**;
- **ownership não se infere do silêncio** de outro agente;
- um terminal só começa Story nova quando a Lane 0 **registrar a atribuição**;
- um agente **ativo** que detecta o evento que lhe corresponde **continua sem pedir autorização**;
- um agente **encerrado** só volta por um mecanismo externo de orquestração;
- hooks podem **notificar**, mas **não concedem ownership**;
- automação de dispatch **não contorna** gates, anticolisão nem separação de Writers.

### Fila de revisão (QA)

Com mais de um PR aguardando: **1)** Critical/High e migrations; **2)** caminho crítico P0; **3)** PR mais próximo da integração; **4)** baixo risco.

**O CI roda antes da revisão.** Revisor não gasta atenção em PR que ainda falha em check mecânico — lint e typecheck a máquina encontra sozinha, e o custo real do revisor é a análise que ela não faz.

### Decisões técnicas

Havendo mais de uma solução válida, escolha autonomamente a que respeita PRD e Arquitetura, reutiliza padrões existentes, produz a **menor mudança correta**, é reversível, preserva compatibilidade, tem testes, reduz risco de integração e acelera a entrega. Registre decisões relevantes, mas **não espere aprovação para continuar** — e não reabra decisão já aprovada.

### Falhas e correções

Teste, build, revisão ou CI vermelho: investigue → identifique a causa → corrija → adicione regressão quando necessário → reexecute → siga até o gate ficar verde. Não peça autorização para corrigir problema dentro do escopo. Achados **CRITICAL e HIGH** são corrigidos **antes do merge**.

### Foco de execução

Evite relatório extenso durante a implementação, reconstrução de contexto já registrado, confirmação para decisão reversível, refatoração ampla, melhoria cosmética sem impacto P0, abstração nova sem necessidade, dependência ou upgrade amplo sem justificativa, e marcar Story como done sem evidência no `main`.

Prefira mudanças verticais pequenas, padrões existentes, testes direcionados durante o desenvolvimento, suíte proporcional ao risco antes do PR, suíte completa nos gates de integração, correção imediata, checkpoints curtos e avanço automático.

### Regra de merge e closure

Ordem obrigatória: implementação → **todos** os testes obrigatórios → revisão → CI verde → merge da implementação **completa** → confirmação da evidência em `origin/main` → closure → atualização do `sprint-status`.

A Lane 0 — e só ela — libera o merge, e **somente** quando: CI verde; nenhum finding Critical/High aberto; findings obrigatórios resolvidos; migration **e rollback comprovados** quando aplicável; integração com `origin/main` atualizada; ownership e escopo íntegros.

**Nunca faça closure enquanto código ou teste obrigatório estiver apenas em outra branch ou na fila de merge.** O closure só ocorre depois de **todos** os PRs obrigatórios da Story estarem no `main`. Havendo múltiplos PRs dependentes, coordene explicitamente a ordem.

**Nenhum comentário isolado é trava técnica de merge.** Bloqueio se expressa por gate verificável, check obrigatório ou status operacional explícito — um bloqueio que só existe na prosa de um comentário não sobrevive a uma troca de sessão.

### Gates que não podem ser cortados

Nunca reduzir nem eliminar: isolamento multi-tenant, RLS, autorização, testes negativos, segurança, LGPD, migrations seguras, rollback, backup, restauração, observabilidade, health checks, smoke tests e evidências de aceite.

**P0 não pode ser escondido por feature flag** — flags só para P1 não essencial.

### Condições de parada

Pare e peça decisão **somente** diante de: dois Writers na mesma Story ou worktree; conflito capaz de apagar trabalho; migration destrutiva ou irreversível; risco real de perda de dados; secret ou credencial indispensável indisponível; decisão de produto que altere materialmente o contrato de dados; vulnerabilidade CRITICAL/HIGH sem mitigação segura; indisponibilidade externa que bloqueie todo o caminho; aprovação final para produção.

Antes de parar: preserve o estado → registre o bloqueio → **continue todas as lanes independentes** → apresente recomendação objetiva → ofereça no máximo duas opções, destacando a recomendada.

### Continuidade

Ao concluir uma Story: merge e closure → atualizar status e checkpoint → **a Lane 0** identifica a próxima Story P0 elegível e atribui ownership exclusivo → o agente atribuído continua automaticamente. **Não espere o usuário mandar "continue"** — mas também não se autoatribua: a atribuição é da Lane 0 (ver **Papéis**).

**Continue automaticamente** quando as quatro condições valerem juntas: a próxima ação está na sua alçada; as dependências estão satisfeitas; o ownership está confirmado; não há colisão nem condição material de parada. Não peça autorização para testar, corrigir lint/formatação, integrar `origin/main`, responder findings, atualizar o próprio PR, acompanhar CI, preparar evidências ou fazer deploy em staging já autorizado por este protocolo.

Cada sessão tem um **nome estável** ligado ao papel ou à Story, para que a retomada seja endereçável.

Antes de compactar ou encerrar uma sessão, registre: SHA do `main`; branch, worktree e owner; PR, CI e testes; alterações não publicadas; decisões e bloqueios; a **próxima ação exata** e a instrução de retomada.

**Ao retomar:** restaure a sessão nomeada → `git fetch` → releia PR, CI e board → revalide ownership → inspecione worktree e alterações existentes → continue do último estado **real confirmado**. O checkpoint orienta a retomada, **não a substitui**: ele descreve o que era verdade quando foi escrito, e uma queda costuma acontecer exatamente entre o checkpoint e o fato.

## Protocolo Autônomo Fast Track por Épicos

Camada de **Épico** sobre o **Protocolo Autônomo de Aceleração** (acima), que é a camada de **Story**. O que já está definido lá — fonte de verdade, papéis, ownership, paralelismo, barramento, board, roteamento, autonomia, condições de parada e continuidade — **não se repete aqui**; esta seção acrescenta a ordenação por Épico, a **proporcionalidade dos gates ao risco** e a regra que impede validação duplicada. Havendo conflito material entre as duas camadas, **prevalece a regra mais segura**.

O objetivo é acelerar sem enfraquecer autenticação, autorização, RLS e isolamento multi-tenant, integridade e migrations, atomicidade, concorrência, idempotência, proteção de dados e segredos, observabilidade, segurança, LGPD, smoke real de integrações críticas, proteção do SHA integrado e auditoria consolidada antes da produção.

### Contrato entre as fontes

PRD, UX, Arquitetura e Constitution definem o **contrato do produto**; a especificação da Story define o **recorte executável**; `origin/main`, GitHub, PRs e CI são o **estado técnico real**; `sprint-status` registra o **andamento**. Este arquivo governa o **processo de execução** e **não substitui contrato funcional**. A ordem de reconciliação é a de **Fonte de verdade** (acima).

Em divergência material entre fontes, **não invente requisito**: reconcilie as fontes ou registre a decisão pendente e siga pelo trabalho que não depende dela.

### Abertura do Épico

1. reconciliar `origin/main`, GitHub, `sprint-status` e dependências;
2. identificar Stories concluídas, ativas, bloqueadas e prontas;
3. montar **uma única** ordem de execução derivada das dependências;
4. classificar antecipadamente o **risco** de cada Story (abaixo);
5. definir ownership de cada branch e worktree — atribuição é da **Lane 0** (ver **Papéis**);
6. iniciar imediatamente a primeira Story **realmente** executável.

Encerrada e integrada uma Story: atualizar a fonte de status → liberar o Writer anterior → selecionar a próxima Story pronta → iniciar **sem pedir nova autorização ao dono** → seguir até o fim do Épico ou até um bloqueio real. "Sem nova autorização" é sobre o **dono**, não sobre ownership: quem atribui a Story continua sendo a Lane 0, e nenhum agente se autoatribui.

Não pare entre passos normais só para pedir confirmação, reapresentar plano já autorizado, pedir permissão para testar, ou perguntar se deve iniciar a próxima Story pronta. Commit, push, PR e merge permitidos pelo protocolo já estão autorizados (ver **Autonomia**).

### Ciclo da Story dentro do Épico

O percurso é o de **Fluxo obrigatório** (acima). O Fast Track acrescenta apenas:

- a **classificação de risco** acontece antes da implementação e determina quais gates se aplicam;
- correção de finding é do **mesmo Writer**, no mesmo PR, e a **re-QA é focada**: novo diff, findings reportados e regressões plausíveis;
- o merge é protegido pelo **SHA exato aprovado**, com CI verde no `main` depois dele;
- `DONE` exige critérios de aceite, integração, CI e closure **comprovados** — nunca a ausência de evidência contrária.

### Spec Kit consolidado

Quando a Story já estiver suficientemente definida, execute em **uma passagem contínua**: `specify` → `clarify` → `plan` → `checklist` → `tasks` → `analyze` → implementação → `converge`. Sem pausas artificiais entre etapas — o que não pode desaparecer é a **evidência** de cada uma.

Interrompa **antes** da implementação apenas diante de: decisão material do dono; conflito entre PRD, UX e Arquitetura; requisito realmente ambíguo; operação irreversível não autorizada; dependência externa sem fallback; colisão de Writer; risco de expandir materialmente o escopo.

**Correção de QA não reinicia o Spec Kit.** Ela exige reprodução, remediação mínima, teste de regressão, os gates afetados e re-QA focada.

### Classificação de risco e gates proporcionais

Proporcionalidade decide **quais gates se aplicam**, nunca autoriza cortar gate aplicável — os de **Gates que não podem ser cortados** e de **Processo obrigatório** continuam valendo integralmente.

**Risco baixo** — documentação, testes sem alteração de produção, copy, UI isolada, refatoração comprovadamente sem mudança comportamental. Gates: revisão do diff; testes diretamente afetados; lint/typecheck incremental quando aplicável; CI mínimo do repositório.

**Risco médio** — regra de negócio, endpoint, DTO, service/controller, automação, integração interna sem alteração estrutural de segurança. Gates: testes afetados; integração da área modificada; typecheck; lint; build; CI obrigatório; QA focada nos critérios de aceite e nas regressões prováveis.

**Risco alto** — autenticação, autorização, RLS, multi-tenancy, permissões, migration, concorrência, idempotência, arquivos, segredos, pagamentos, exclusão ou operação destrutiva, alteração de contrato público. Gates: testes específicos da área crítica; **integração real** (banco de verdade, nunca mock); regressão de segurança relacionada; typecheck, lint e build; **migration drill e rollback** quando houver migration; QA cruzada; CI obrigatório **no SHA exato**; validação pós-merge no `main`.

Uma mudança que **toque** área crítica é **automaticamente elevada** ao risco alto, independente da classificação inicial.

### Sem gate duplicado

- O **Writer** prova a implementação e executa os testes afetados; o **CI** executa os checks canônicos; a **QA** reproduz comportamentos críticos, critérios de aceite e findings.
- A QA **não repete mecanicamente** a execução do Writer — o custo real do revisor é a análise que a máquina não faz.
- **Não reexecute a mesma suíte no mesmo SHA sem evidência nova.** Um novo SHA invalida apenas as evidências das áreas **materialmente afetadas**; correção pequena recebe re-QA concentrada.
- Auditoria completa da Story só se reabre quando contrato, arquitetura ou área crítica mudou.
- A **suíte completa** é prioritária: em mudança de alto risco que a exija, no encerramento do Épico, periodicamente no `main`, antes de staging e antes de produção.

### Dependência externa e EXTERNAL_GATE

Dependência externa indisponível **não bloqueia trabalho independente**. Com o código integrado e faltando só a validação externa: registre um `EXTERNAL_GATE` com responsável, condição de desbloqueio e teste pendente; **não marque `DONE`** se o aceite exigir o smoke externo; siga nas Stories independentes; execute o smoke assim que a credencial ou o serviço existir.

Nunca substitua a dependência por implementação falsa, mock silencioso ou evidência de sucesso inexistente.

### Encerramento do Épico

1. suíte completa e testes de integração do Épico;
2. validação dos contratos **entre** as Stories;
3. revisão de migrations e rollback aplicáveis;
4. segurança, RLS, LGPD e observabilidade, proporcionalmente às áreas alteradas;
5. reconciliação de débitos técnicos e `EXTERNAL_GATE`s;
6. documentação e status atualizados;
7. CI verde no `main`;
8. relatório consolidado do Épico publicado;
9. início automático do próximo Épico autorizado.

### Anti-estagnação e monitores

**No máximo um monitor por evento**, sem duplicatas e sem rearmar indefinidamente antes de confirmar o handoff. Monitor expirado sem avanço: reconcilie o estado real em vez de esperar de novo. Dependendo do Writer, entregue findings **executáveis** e registre `BLOCKED_BY_WRITER` — evento operacional que a Lane 0 trata como os demais de **Roteamento determinístico**. Havendo Story independente pronta, o Épico continua em vez de parar inteiro.

Relatório antigo sobre o **mesmo SHA** não é evento novo.

### Relatório operacional

Concluída uma ação relevante, informe objetivamente: Épico e Story; estado real; ownership; classificação de risco; gates selecionados; evidências e CI; findings ou débitos; `EXTERNAL_GATE`s; próxima ação automática. Sem repetir relatório, replanejar o que já foi executado ou pedir confirmação desnecessária.

## Invariantes conceituais (nunca erodir)

`Pipe ≠ Database` · `Card ≠ Registro` · `Fase ≠ Status do Card` · `Super Admin (Plataforma) ≠ Admin da Organização` · `Usuário ≠ Organização`. Identidade é Account global + Membership por Organização. Autorização é **deny-by-default** e `PERMISSÃO = AÇÃO + ESCOPO`; isolamento por Organização é o invariante-mãe. Os três Formulários (inicial, de fase, de database) são independentes.

## Arquitetura

Monorepo pnpm workspaces (`apps/*`), TypeScript estrito compartilhado via `tsconfig.base.json` (`strict` + `noUncheckedIndexedAccess`). O frontend consome apenas a API interna e nenhuma superfície de API pública é implementada. Nenhuma regra de domínio deve existir no frontend.

**Estado transitório da implementação — Story 3.8 (Épico 3: Campo Arquivo funcional e anexo geral).** O L1 está fechado: existe banco com isolamento por RLS, autenticação e sessão (1.4/1.5), contexto de Organização resolvido no servidor (1.3), autorização por papel (1.6, CASL deny-by-default) e a casca navegável (1.7/1.8). O Épico 2 avançou: **existe `Pipe`** (2.1 — ciclo de vida/catálogo, Admin da Org), **`PipeGrant`** (2.2 — concessão de papel por Pipe + acesso por concessão), **`Phase`** (2.3 — Fase do fluxo: criar/renomear/reordenar/arquivar/restaurar, ordenação fracionária, invariante "≥1 Fase ativa") e o **domínio Formulário** (2.4): **catálogo canônico de tipos de Campo** (enum `FieldType`, 12 tipos, de plataforma), **`Form`** (Formulário de um contexto — `PIPE_INITIAL`/`PHASE`; `DATABASE` é contrato do E3, sem owner ainda) e **`Field`** (Campo — rótulo/tipo/ajuda/`typeConfig` JSON/valor padrão/posição fracionária/estado). Montagem = obter (getOrCreate lazy no adicionar; ler não cria), adicionar e reordenar Campo; contextos inicial e de Fase **independentes** (INV-FORM-01). Opções de Seleção vivem no `typeConfig` (JSON com UUID estável), não em tabela (DBT-2.4-OPCOES-JSON). O tipo Arquivo é gated (AD-28, fail-closed): `FILE_UPLOAD_ENABLED` default falso + função pura `podePublicarComArquivo` (contrato consumido pela publicação, 2.6 — DBT-2.4-FILE-GATE-CONSUMO). A **evolução segura de Campos** (2.5) fecha o L2 de configuração: editar rótulo/ajuda/valor padrão (não `type`, imutável), arquivar/restaurar Campo (transição reversível; restaura ao **final** da ordem ativa) e o **ciclo de opções de Seleção** (adicionar/renomear/reordenar/arquivar/remover). As opções seguem em JSON no `typeConfig` (Opção A — **sem migration**; `state`/`archivedAt` já existiam em `Field` desde a 2.4). `option-config.ts` é o núcleo puro dos invariantes das opções (id estável/único, `label` não confiável, allowlist de chaves, fail-closed, limites); o serviço reescreve o `typeConfig` inteiro num único `update` com **guarda otimista** (`typeConfig: { equals: <lido> }` → **409** em conflito, sem lost update silencioso). Sem exclusão: remover opção é UPDATE, nunca DELETE de linha. A **publicação** (2.6) fecha o ciclo do Formulário: publicar congela o rascunho **validado** numa **`FormVersion`** imutável (snapshot JSON integral dos Campos ativos, numerada por `@@unique([orgId, formId, version])`); despublicar zera o ponteiro `Form.publishedVersion` (preserva o histórico); ler devolve estado e snapshot. **Imutabilidade é do banco**: o runtime tem GRANT só `SELECT`/`INSERT` em `FormVersion` (sem UPDATE/DELETE). Validação de publicabilidade em `snapshot.ts` (fail-closed): sem Campo ativo, Seleção sem opção ativa, gate de Arquivo (`podePublicarComArquivo`) e `typeConfig` malformado → 400; o snapshot **não inventa obrigatoriedade** (não existe em `Field`). A publicação toca duas escritas (INSERT versão + UPDATE ponteiro) numa **transação interativa com contexto no client raiz** — o `withTenantContext` recusa `$transaction` no client estendido, mas o raiz roda o mesmo primitivo `set_config(..., true)`, agora com **fonte única** (`definirContextoOrg` em `tenant-context.ts`); concorrência de número resolve por `UNIQUE` → rollback integral → 409 (P2002/P2028). A **submissão interna** (2.7) abre o ciclo do Card: submeter o Formulário inicial **publicado** de um Pipe cria um **`Card`** na 1ª Fase ativa (menor `position`), que referencia a `FormVersion` publicada no ato (definição **congelada** — AD-12) e guarda os `valores` validados contra o snapshot em **JSONB por `Field.id`** (opção por `id`, nunca rótulo; sem tabela de valores por Campo — AD-11). Um evento **`CardHistory`** `CREATED` é escrito na **mesma transação** interativa (AD-13; mesmo primitivo `definirContextoOrg` da 2.6): não há Card sem evento. Validação em `submission.ts` (fail-closed): allowlist anti-mass-assignment (chave desconhecida → 400), tipo por Campo, Seleção por `id`; valor ausente é permitido (não há obrigatoriedade em `Field`). **Idempotência** por `@@unique([orgId, formId, idempotencyKey])` — retry devolve o Card existente; o conflito reconhece **P2002 e P2028** → idempotente ou 409, **nunca 500**. **GRANT como fronteira**: `Card` tem só `SELECT`/`INSERT` (a 2.7 apenas cria; UPDATE/DELETE ficam para a movimentação/ciclo de vida do Card — 2.14/2.11 — com consumidor e teste que provam o escopo) e `CardHistory` só `SELECT`/`INSERT` (append-only imutável, como `FormVersion`). A **submissão pública controlada** (2.8) abre o canal externo do Formulário inicial: um endpoint **não autenticado** (`POST /public/forms/:publicId/submit`) resolve o tenant por um **`PublicFormRoute` GLOBAL** (sem RLS por definição — AD-10, análogo a `Account`) que mapeia um `publicId` **opaco** para `(orgId, formId)`, entra em `withTenantContext(orgId)` e **RELÊ o Form sob RLS** — nunca confia em `orgId`/`formId`/`pipeId`/`formVersionId` do cliente; link inválido/revogado/não publicado/cross-tenant → **404 uniforme**; a resposta é só `{ ok: true }` (sem vazamento). A submissão vira uma **`SubmissaoPublica`** org-scoped com **ciclo próprio** (`PENDING`/`CONVERTED`/`REJECTED`) — **`Fase ≠ Status do Card`** preservado. **TRIAGE** (padrão) fica PENDING até a triagem; **DIRECT** converte na hora. Na triagem, **aprovar = converter é atômico** (AD-11): cria **1 Card** (`origin=PUBLIC`) na 1ª Fase ativa + evento `CREATED` na mesma transação raiz; **rejeitar** preserva (`REJECTED`, sem Card). A conversão trata **P2002 e P2028** → idempotente/409, **nunca 500**, com auditoria manual (FR-214). A autorização de revisão é por **capacidade explícita `reviewPublicSubmissions`** na `PipeGrant` (deny-by-default; Admin da Org implícito; demais só por concessão), reusando a guarda fina de `pipe-authz.ts` (`exigirRevisarSubmissoesPublicas`, C3 congelado). Baseline antiabuso: **rate limit atômico** por IP confiável + `publicId` (fail-closed → 429), **Arquivo bloqueado no público** (AD-28), sem CAPTCHA. **GRANT como fronteira**: `SubmissaoPublica` tem `SELECT`/`INSERT`/`UPDATE` — **sem DELETE** (preserva o dado do titular — LGPD; rejeitar é `state`); `PublicFormRoute` idem **sem DELETE** (revogar é `active=false`; rotacionar = revogar + criar novo `publicId`). O **Kanban e o espaço operacional do Card** (2.9) abrem a superfície de **LEITURA** sobre `Card`/`Phase` já materializados — **sem migration e sem GRANT novo** (o runtime segue sem UPDATE em `Card`; **mover é a 2.14**). `KanbanReadService` reusa `resolverPoderNoPipe` (basta **qualquer** poder no Pipe — inclusive **VIEWER concedido**, pois ler ≠ operar; sem acesso → 404 não-enumerante) e devolve as **capacidades** derivadas do poder do próprio principal (nunca revela ação alheia). `verKanban` lista as colunas (Fases ativas por `position`) com a contagem via **um `groupBy`** (sem N+1); a coluna **pagina por cursor determinístico** `[createdAt, id]` (teto 100 — NFR-3/4). `valores` (possível PII) só no **detalhe** do Card, nunca na lista nem em log; `orgId` fora da fronteira. A divergência "2.9 = movimentação" (comentário legado da migration da 2.7) foi resolvida a favor do `epics.md` (movimentação = 2.14); o design de movimentação vive no Apêndice A da spec da 2.9. O **acesso, Responsável e concessões de Card** (2.10) fecha o **DBT-2.2-ROLE-DORMENTE**: introduz a autorização **no nível do Card** sobre o substrato de Pipe, com três dados novos org-scoped. **`CardGrant`** (D-OA1) é a concessão DIRETA de acesso a UM Card (Observador = ler; operacional = operar [+`podeMover` — só o DADO da capacidade; a operação é a 2.14]); RLS+FORCE+WITH CHECK, GRANT `SELECT/INSERT/UPDATE` **sem DELETE** (revogar = `state`). **`CardResponsavel`** (D-OA2) é a atribuição de Responsável por Card, chave `membershipId`; mantém `Card` **append-only** — a 2.10 **não** abre GRANT de UPDATE em `Card` (o 1º UPDATE segue reservado à movimentação, 2.14; invariante preservado); `creator` continua sendo o `actorId` do `CREATED` (2.7), sem coluna nova. **`PipeGrant.restritoAoProprio`** é um modificador do Membro (novo DADO de autorização, análogo a `reviewPublicSubmissions` da 2.8; **não** é papel novo), exposto pelas rotas de grant. A resolução `resolverAcessoNoCard` (em `pipe-authz`) **compõe** papel-de-Pipe + concessão direta + "restrito ao próprio" + Responsável-atual: deny-by-default, sem acesso → **404 não-enumerante**, ler-sem-operar → **403**; **creator e histórico NUNCA concedem acesso** (SC-2105). Atribuir/remover Responsável exige **operar o Card** (`exigirOperarCard`, mais restrito que operar o Pipe — um Membro "restrito ao próprio" não mexe em Cards que não acessa) e o **alvo** precisa de acesso operacional prévio (SC-2101; atribuir não amplia acesso — SC-2102); conceder/revogar/listar exige **gerenciar o Pipe** (`exigirGerenciarPipe`). Cada mutação escreve um evento `CardHistory` (`RESPONSAVEL_ASSIGNED`/`CHANGED`/`REMOVED`, `ACCESS_GRANTED`/`ACCESS_REVOKED`) na **mesma transação** interativa no client raiz (`definirContextoOrg`, auditoria manual FR-214). Idempotência por índice PARCIAL de unicidade ativa (1 Responsável ativo por Card; 1 concessão ativa por (Card, pessoa)); conflito reconhece **P2002 e P2028** → 409, nunca 500. O **contrato de Membership** (D-OA3) é materializado como **funções PURAS** (`membership-contract.ts`: preflight de encerramento + handler pós-alteração), **sem chamador** — o E8 as consumirá; o preflight nasce **vacuamente verdadeiro** (a regra "Card exige Responsável ativo" não existe para Card na Fase 1 — DIV-3, não inventada); reativar não restaura acesso; `creator` preservado por construção. O **ciclo de vida do Card** (2.11) introduz o eixo de estado — `ATIVO`/`FINALIZADO`/`ARQUIVADO` (`reaberto`/`restaurado` são **transições**, não estados) — independente da Fase (`Fase ≠ Status do Card`) e da saúde temporal (2.13). Finalizar/reabrir/arquivar/restaurar são transições **atômicas**, **idempotentes** e **auditadas** (evento próprio no `CardHistory`: `FINALIZED`/`REOPENED`/`ARCHIVED`/`RESTORED`, na mesma transação interativa — `definirContextoOrg`, auditoria manual FR-214). O **estado anterior ao arquivamento** vive em `Card.previousLifecycleState` e a restauração o devolve (FINALIZADO→arquivar→restaurar volta a FINALIZADO, não a ATIVO). A decisão de QUAL transição é o núcleo **puro** `planejarTransicao`; o serviço a aplica com **guarda otimista** (`updateMany where lifecycleState = <lido>` → reconsulta → idempotente/**409**; P2002/P2028 → 409, nunca 500); transição inválida a partir do estado atual → **409**. Autorização: **operar o Card** (`exigirOperarCard`, 2.10) — 404 sem acesso, 403 só-lê. **O ponto arquitetural:** o ciclo de vida é o **1º UPDATE de `Card`** em runtime, e ele é **column-scoped** — `GRANT UPDATE ("lifecycleState", "previousLifecycleState", "updatedAt") ON "Card"`, e **só** isso; `phaseId` (movimentação, 2.14), `valores` e `orgId` seguem **sem** GRANT de UPDATE (tentativa bate em `permission denied` — provado no `card-lifecycle-rls`), reconciliando "evoluir estado sim, mover/reescrever não" exatamente como a migration da 2.7 antecipou. O detalhe do Card (2.9) passa a expor `lifecycleState` (a 2.13 consumirá); a **re-filtragem da LISTA** do Kanban por estado é da 2.13, não desta Story. Os **marcos por Fase e o override por Card** (2.12) introduzem o eixo TEMPORAL — distinto do ciclo de vida (2.11) e da Fase (`Fase ≠ Status do Card`). Cada **Fase** configura três marcos como **durações relativas à entrada, em minutos** (`Int`, colunas `expected/due/expirationDurationMin` — o Prisma 6.19.3 não tem `interval` nativo; `esperado ≤ vencimento ≤ expiração` validado no núcleo puro **e** por CHECK) e, por marco, um **override absoluto** via `Field.id` de um Campo `DATE`/`DATETIME` do Formulário inicial (`expected/due/expirationFieldId`). Configurar é **"config do Pipe"** (`exigirGerenciarPipe`; Membro→403, sem acesso→404); ler config = `resolverPoderNoPipe`. A **referência temporal de entrada** é a `CardPhaseEntry` org-scoped, **append-only e IMUTÁVEL** (GRANT só `SELECT`/`INSERT`, como `CardHistory`/`FormVersion`): cada entrada preserva o **instante** (`enteredAt` em `@db.Timestamptz` — instante absoluto, diverge do `TIMESTAMP(3)` do schema por decisão de Arquitetura, DIV-1) e a **origem** (`SUBMISSION`/`MOVE`); reentrada é **novo INSERT** (a atual = mais recente por `enteredAt`). O **`configSnapshot`** congela a config vigente da Fase no instante da entrada (decisão do dono **D-OA1=A**): mudar a config afeta **só entradas futuras** — "sem recálculo retroativo silencioso" (epics §945/§951) cai **por construção** (padrão da `FormVersion`). O write-side é o helper compartilhado **`registrarEntradaNaFase(tx, contexto, …)`**, que grava a 1ª entrada (`origin=SUBMISSION`) na **mesma transação** dos DOIS sítios de criação de Card (submissão interna 2.7 e conversão pública 2.8 — não há Card sem sua entrada, AD-13); o mesmo helper é o **contrato de reentrada** (`origin=MOVE`) que a 2.14 consumirá (AD-11 — consumidor concreto agora, movimentação depois). A migração faz **backfill idempotente** da 1ª entrada de cada Card existente, rodando **antes** de `FORCE ROW LEVEL SECURITY` (sob FORCE, o próprio dono seria barrado por `current_org_id()` NULL na migração). O **cálculo dos marcos** (`calcularMarcos`, núcleo puro) é **sob demanda na leitura** (sem agendador — decisão de Arquitetura): precedência **valor-do-Card (override) › duração-da-Fase › ausência** (ausência do valor é ignorada e cai para a duração; valor malformado é fail-closed, idem); a leitura da base vive em `GET /cards/:cardId/phase-entry` (`exigirLerCard`). A 2.12 entrega só a **base** (instantes); o **veredito de saúde** (`ok`/`atrasado`/`vencido`/`expirado`) é a 2.13. `Card` segue **append-only** — a 2.12 **não** abre GRANT de UPDATE em `Card` (o 1º UPDATE column-scoped segue sendo o de ciclo de vida da 2.11; `phaseId`/movimentação sem UPDATE até 2.14). A **saúde temporal do Card** (2.13) fecha o eixo temporal do L2: `ok`/`atrasado`/`vencido`/`expirado` é **derivação PURA na leitura** dos marcos que a 2.12 materializou (`derivarSaude`, núcleo `cards/health/card-health.core.ts`; precedência interna expirado › vencido › atrasado › ok) — **sem persistir, sem evento, sem agendador** (AD-11; coerente com o Kanban só-leitura da 2.9 e o "sem agendador" da 2.12), então `Card` segue **append-only** (a 2.13 **não** abre migration nem GRANT). A precedência de **apresentação** é a função pura `indicadorDominante(lifecycleState, saude)` (ARQUIVADO→`arquivado`, FINALIZADO→`finalizado`, ATIVO→a saúde): os **dois eixos canônicos** (ciclo de vida + saúde) nunca são fundidos num estado persistido — o indicador é só apresentação, e sob ciclo inativo o **ciclo vence** a saúde (que volta a ser derivada ao reabrir/restaurar, automático por ser pura na leitura). Exposta no **detalhe** do Card (`verCard`, 2.9 — campos `saude`/`indicadorDominante` no `CardDetalheVisao`), **nunca na lista**: a saúde com override depende de `valores` (possível PII), que vive só no detalhe (NFR-3/4). A **re-filtragem da LISTA** do Kanban por estado, os **badges/priorização no Dashboard** e a **emissão de evento/persistência** de saúde ficam para o **E7** (consumidor concreto futuro — AD-11), corrigindo a antecipação "2.13" da nota da 2.11.

**O papel de Pipe deixou de ser dormente para _config_.** Na 2.2 o `PipeRole` era armazenado mas inerte (decisão SC-222=B). Na 2.3, **gerenciar Fases é "config do Pipe"** (PRD §7) e ativou o diferencial; na 2.4 **configurar Formulário (inicial e de Fase) também é config do Pipe** (D3.2) e na 2.5 **evoluir Campos (editar/arquivar/opções)** e na 2.6 **publicar/despublicar Formulário idem** — todas **reusam a mesma resolução**: gerencia o **Admin da Org** (qualquer Pipe) **ou** o **Admin do Pipe** (concessão `role=ADMIN` ACTIVE, com `Membership` ACTIVE); MEMBER/VIEWER concedidos só leem. Essa resolução foi **extraída** de `PhasesService` para o helper compartilhado `pipes/pipe-authz.ts` (`resolverPoderNoPipe`/`exigirGerenciarPipe`), consumido por Fases, Formulários, evolução de Campos e publicação — uma única guarda fina (DBT-AUTHZ-01), sem tocar o guard/`ability.ts` (C3 congelado). O Formulário de Fase e seus Campos resolvem o poder pelo Pipe dono da Fase (`phase.pipeId`). Os localizadores de Formulário foram extraídos para `forms/form-locate.ts`, compartilhados por `FormsService` (montagem, 2.4), `FieldsService` (evolução, 2.5) e `FormPublicationService` (publicação, 2.6). Na **2.7 o papel deixou de ser dormente para _operação_**: submeter o Formulário inicial exige **operar** o Pipe (`exigirOperarPipe`, `Poder = gerenciar > operar > ler`) — Admin da Org e **Membro do Pipe** submetem (poder do Membro **ativado**), Viewer concedido só lê (403), sem acesso 404. Na **2.8** a `PipeGrant` ganhou uma **capacidade** (não um papel): `reviewPublicSubmissions` habilita **revisar submissões públicas** (`exigirRevisarSubmissoesPublicas`), deny-by-default e concedida explicitamente — Admin da Org implícito; a config do acesso público (habilitar/revogar/rotacionar) reusa `exigirGerenciarPipe`. **O Épico 3 abre o domínio Database.** O **Épico 2 fechou** (2.14 movimentação de Card entre Fases; 2.15 Formulário de Fase e bloqueio de transição; 2.16 evento canônico de movimentação; 2.17 histórico do Card; 2.18 Pipes relacionados no Perfil — o narrativo detalhado por Story vive nas specs). A **Story 3.1** materializa **`Database`**, a 1ª entidade do E3 — **twin estrutural de `Pipe`**, entidade **DISTINTA** (`Database ≠ Pipe` — RN-061): tabela/enum (`DatabaseState { ACTIVE, ARCHIVED }`)/subject CASL/módulo próprios em `apps/api/src/databases/`, **sem reusar** as entidades de Pipe. Ciclo de vida e catálogo org-scoped: **criar** (201), **listar/obter**, **renomear** e **arquivar/restaurar** (200; idempotentes — o caminho no-op **não** emite `updateMany`, para não registrar um falso `denied` na auditoria). **Somente-leitura integral sob arquivamento (D1):** renomear um Database `ARCHIVED` → **409** (`{ motivo: 'DATABASE_ARQUIVADO' }`, com defesa em profundidade no `updateMany where state: 'ACTIVE'`); o fluxo é restaurar → renomear → arquivar. Restaurar **preserva identidade** (`id`/`name`). **Isolamento pelo banco:** RLS **ENABLE+FORCE** com `WITH CHECK` no INSERT **e** no UPDATE (simétrica a `Pipe`/`Membership`); toda query por `withTenantContext` (nenhum `where orgId` manual); `orgId` fora do payload e nunca aceito do cliente. **Sem exclusão:** o runtime tem `GRANT SELECT/INSERT/UPDATE` e **não** DELETE — a policy `delete` existe só por simetria/defesa em profundidade, mas quem barra o runtime é o GRANT (uma rota de DELETE por engano bateria em `permission denied`). **Autorização (3.1):** só o **Admin da Org** alcança (`ler`/`administrar` Database via CASL; MEMBER/GUEST → 403, deny-by-default), sem tocar o guard/`ability.ts` (C3 congelado); papéis **finos por Database** são a **3.2**. O núcleo **puro** `database-lifecycle.ts` (`planejarArquivamento`/`planejarRestauracao`/`podeEditarDatabase`) espelha `card-lifecycle.transitions.ts`. **Não existe ainda no E3:** Registros, Views e a **somente-leitura integral sobre dados dependentes** — contrato futuro (3.3/3.4/3.7/3.8/3.9); em 3.1 não há dado dependente, então nada é bloqueado além do renomear-sob-arquivamento. A **Story 3.2** abre os **papéis e o acesso por Database** — **twin estrutural de `PipeGrant`** (2.2), entidade **DISTINTA** (`Database ≠ Pipe` — RN-061): a tabela **`DatabaseGrant`** (enums `DatabaseRole { ADMIN, MEMBER, VIEWER }` e `DatabaseGrantState { ACTIVE, REVOKED }`) concede um papel a uma **`Membership`** (o vínculo Account×Org, nunca à `Account` global) num Database. **No máximo um papel ATIVO por (Database, pessoa)** é imposto pelo **banco** via índice único **parcial** `WHERE state='ACTIVE'` (raw SQL — o Prisma 6.19.3 não expressa índice parcial): a 2ª concessão ativa colide no INSERT (**P2002 → 409**), sem leitura-antes-de-escrever. **Sem exclusão:** GRANT `SELECT/INSERT/UPDATE`, **não** DELETE — revogar é soft-delete (`state=REVOKED`, `revokedAt`), re-conceder é uma linha ACTIVE nova. **Isolamento pelo banco:** RLS **ENABLE+FORCE** com `WITH CHECK` no INSERT **e** no UPDATE (simétrica a `PipeGrant`); toda query por `withTenantContext`. **A diferença real frente à 2.2 é a autoridade HIERÁRQUICA de concessão** (D3.4 §969/§1086), resolvida **no serviço** por `exigirConcederPapel` (`database-authz.ts`): **Admin da Org** concede/altera/revoga **qualquer** papel; **Admin do Database** (concessão `role=ADMIN`) concede **só** `MEMBER`/`VIEWER` — tocar `ADMIN` do Database é **só** do Admin da Org (403 senão); Membro/Somente-leitura → 403; sem acesso → **404 não-enumerante**. **Teto da Org (AD-9):** um Convidado (Membership `role=GUEST`) só recebe `VIEWER` (senão → 400). **`ler Database` foi aberto na CASL para a guarda GROSSA** (era Admin-only na 3.1) — MEMBER/GUEST passam o guard e a autoridade **fina** decide no serviço (DBT-AUTHZ-01); `administrar Database` **segue Admin-only** (ciclo de vida da 3.1 + conceder `ADMIN` do Database). **Catálogo não enumera sem acesso (RV-1):** `listar` ramifica — Admin da Org vê **todos**; não-Admin vê **só** os Databases onde tem `DatabaseGrant` ACTIVE; `obter` sem concessão → **404 não-enumerante** (`resolverPoderNoDatabase`). **Admin da Org continua vendo todos e controlando o ciclo de vida (RV-3).** O **poder diferencial MEMBER×VIEWER** sobre Registros/schema segue **dormente** (contrato futuro 3.3/3.4 — não há Registro em 3.2, AD-11; análogo ao SC-222=B da 2.2): a resolução já **distingue** os poderes (`gerenciar`/`operar`/`ler`) para os consumidores futuros e para a autoridade de concessão. **Não existe ainda em 3.2:** Registros, Views e as travas de somente-leitura sobre dados dependentes (contrato futuro 3.3+). A **Story 3.3** ativa o **contexto `DATABASE` do `Form`** (stub desde 2.4) **reutilizando o Form Builder canônico de E2** — montagem (2.4), evolução segura de Campos (2.5) e publicação por `FormVersion` imutável (2.6) — **sem segundo builder nem segundo catálogo** (INV-FORM-01): o Formulário de Database é o **terceiro Formulário independente** (inicial, de Fase, de Database). A única mudança estrutural é o **owner `Form.databaseId`** (migration: coluna + FK Cascade + **DROP+ADD do CHECK `Form_context_owner_ck`** para as 3 cláusulas de contexto↔owner — PIPE_INITIAL/PHASE/DATABASE, cada uma com exatamente seu owner e os demais NULL — + índice único **parcial** `Form_database_uq WHERE context='DATABASE'` + `@@index([orgId, databaseId])`); **sem GRANT novo** (`Form`/`Field` já `SELECT/INSERT/UPDATE` sem DELETE; `FormVersion` só `SELECT/INSERT`, imutável — é o que garante que evoluir o schema **não corrompe Registros** já criados, contrato consumido por 3.4). A **novidade de autorização** é o **roteamento por contexto** (`pipes/forms/form-authz.ts`, DBT-AUTHZ-01): `exigirGerenciarForm`/`resolverPoderNoForm` roteiam por `alvo.databaseId ? database-authz : pipe-authz` — o Formulário de Database resolve o poder por `database-authz` (3.2: **gerenciar** = Admin da Org / Admin do Database; **ler** = qualquer poder no Database; sem acesso → **404 não-enumerante**), enquanto inicial/de Fase seguem por `pipe-authz`. **MEMBER/VIEWER do Database só leem o schema** (403 ao mutar) — o poder diferencial de MEMBER sobre Registros segue **dormente** (3.4). `AlvoFormulario` ganhou `databaseId?` e `form-locate`/`FormsService`/`FieldsService`/`FormPublicationService` foram **generalizados** (aditivo — o caminho Pipe/Fase é o default, regressão de E2 provada verde). As **rotas** são Database-específicas (`databases/:databaseId/form...`, subject `Database`, `@Requer('ler','Database')` grosso), servidas por controllers em `databases/forms/` que **reusam** os serviços do builder (exportados por `PipesModule`, que `DatabasesModule` importa — relação unidirecional, `database-authz` puro, sem ciclo). **Campo Arquivo** segue gated (AD-28): montar é permitido (está no catálogo), publicar é gated (`podePublicarComArquivo` em `snapshot.ts`). **Não existe ainda em 3.3:** criação de Registro, ação `Novo Registro` e submissão (3.4 — a 3.3 entrega só o **schema publicável** e o estado de publicação que 3.4 consumirá); Views/navegação de tabelas (3.5); permissões por Campo (fora da Fase 1). A **Story 3.4** materializa a **1ª entidade de DADO DO TITULAR do Database** — **`Record`** (o **Registro**) — e o **write-side** da sua trilha (**`RecordHistory`**, append-only). **`Card ≠ Registro`** (invariante): entidade **distinta** (`databases/records/`), reusa a **lógica** de submissão (2.7) e ciclo de vida (2.11), **nunca** as entidades de Card; o Registro pertence a **exatamente 1 Database** (RN-063) e **não é transferível**. **Criação idempotente** reusando `submission.ts` (validação pura contra o snapshot da `FormVersion` **publicada** do Formulário de Database, 3.3 — allowlist anti-mass-assignment, tipo por Campo, Seleção por `id`): submeter cria **≤1 Registro** (`origin=NOVO_REGISTRO`) que referencia a versão publicada no ato (**congelada** — AD-12) e guarda os `valores` em **JSONB por `Field.id`** (AD-11); evento `CREATED` na **mesma transação** interativa no client raiz (`definirContextoOrg`, AD-13). **Idempotência** por `@@unique([orgId, databaseId, idempotencyKey])` — retry devolve o Registro existente; conflito reconhece **P2002 e P2028** → idempotente/409, **nunca 500**. **Edição de valores** revalida contra a `FormVersion` **do próprio Registro** (não o rascunho — AD-12), grava `valores`, evento `VALUES_UPDATED`; bloqueada sob Registro/Database arquivado (409). **Ciclo de vida** de **2 estados** (`ATIVO`/`ARQUIVADO` — **sem** `FINALIZADO`, que é do Card): núcleo **puro** `record-lifecycle.transitions.ts` (arquivar/restaurar **idempotentes**, sem estado inválido; restaurar sempre volta a ATIVO, sem `previousLifecycleState`); o serviço aplica com **guarda otimista** (`updateMany where lifecycleState=<lido>`→409) e evento `ARCHIVED`/`RESTORED` na mesma transação; o caminho no-op **não** emite `updateMany` (sem falso `denied` na auditoria). Arquivar **não** é bloqueado por vínculos e preserva tudo; restaurar preserva **identidade/valores/Histórico**. **Autorização:** acorda o poder **dormente** de MEMBER do Database — operar (criar/editar/arquivar/restaurar) = **`exigirOperarDatabase`** (novo em `database-authz`, espelho de `exigirOperarPipe`): Admin da Org / Admin do Database / **MEMBER** operam; **VIEWER** → **403** (só lê); sem acesso → **404 não-enumerante**. Ler o Registro = `exigirLerDatabase`. Guard C3 congelado. **GRANT como fronteira:** `Record` tem `SELECT/INSERT` + **UPDATE column-scoped** de **só** `lifecycleState`/`valores`/`updatedAt` (ciclo de vida + edição) e **sem DELETE** (sem exclusão física — LGPD; arquivar é `state`); `databaseId`/`formVersionId`/`orgId`/`origin`/`idempotencyKey` seguem **sem** UPDATE ("não transferível" + "definição congelada" garantidos pelo banco — tentativa bate em `permission denied`, provado no `records-rls`). `RecordHistory` tem **só `SELECT/INSERT`** (append-only imutável, como `CardHistory`/`FormVersion`). Ambas RLS **ENABLE+FORCE** + `WITH CHECK` no INSERT e UPDATE, em `MODELOS_AUDITADOS`. A leitura é só o **detalhe** (`GET /records/:recordId`, `valores` possível PII fora de log/lista). **Não existe ainda em 3.4:** visualização/tabela/navegação/filtros e a UI `Novo Registro` (3.5, INV-REPORT-01); **read-side** do Histórico (timeline/projeção/mascaramento — 3.6); Campo Arquivo funcional/anexos/eventos de arquivo (3.7/3.8, gate AD-28); vínculo Card↔Registro e `correlationId` (3.9); ação de Automação `Criar Registro relacionado` (E4 — consumidor futuro do mesmo endpoint de criação, AD-11). A **Story 3.5** abre a superfície de **LEITURA** (tabela/navegação) sobre `Record` — **sem migration e sem GRANT novo** (o runtime já lê via `SELECT`), espelhando o **Kanban read (2.9)** no rigor (autz por acesso de leitura, projeção controlada, `orgId` fora da fronteira), com a **diferença** de que o **Registro É o dado**: a tabela **exibe `valores`** (não há PII por-linha a esconder como no Card, cujo acesso é por-Card; o acesso ao Registro é **por Database**). `RecordsReadService.listar` (`GET /databases/:databaseId/records`) exige **`exigirLerDatabase`** (qualquer poder — ler ≠ operar; sem acesso → **404 não-enumerante**), devolve `RecordPaginaVisao` (linhas com `valores`/`lifecycleState`/**`podeEditar`** derivado + `total` + `colunas` = Campos ativos), **ATIVOS por padrão** (opção autorizada `incluirArquivados`), **paginação offset** (`take ≤ 100`, `skip`) + `total` escopado. **INV-REPORT-01 cai por construção:** o acesso a Registro é **por Database** (não há grant por-Registro), então a listagem é sempre escopada a UM Database legível e a contagem só conta os visíveis dele — sem agregação cross-Database. **Filtros mínimos por tipo** (texto contém/igual; número/data igual/maior/menor/intervalo; Seleção contém opção; Sim/Não; combinação por **E**) e **ordenação por Campo** operam sobre `valores` JSONB (por `Field.id`). Como o Prisma 6.19.3 **não** expressa `orderBy` sobre path JSON, a listagem é **uma query raw parametrizada** (`Prisma.sql`) rodada **sob RLS** pelo mesmo primitivo das escritas — `$transaction([...definirContextoOrg, $queryRaw(linhas), $queryRaw(total)])` (o `withTenantContext` embrulha operações de modelo, não `$queryRaw`). **Segurança da query:** o núcleo **puro** `record-query.core.ts` valida a query contra a definição (**allowlist** de `Field.id` + operadores por tipo, coerção/validação de valor por tipo, **fail-closed** → 400), e o SQL é **totalmente parametrizado** (Campo/operador/valor **nunca** concatenados — provado por teste de injeção: valor com metacaracteres SQL é literal). Comparação de **data** é lexicográfica (ISO — evita DoS de cast de valor malformado); **número** usa `::numeric` (seguro — validado como número na escrita). O **Campo Arquivo (`FILE`)** é **gated** no filtro/ordenação (rejeitado → 400) até 3.7/3.8 (AD-28), sem simulação. `podeEditar` **reflete** (não executa) a capacidade de edição (Database ATIVO && Registro ATIVO) — a mutação segue sendo 3.4 (409 sob arquivamento). Guard C3 congelado. **Não existe ainda em 3.5:** grupos E/OU complexos, filtros salvos, visualizações personalizadas, fórmulas, agregações avançadas (fora da Fase 1); read-side do Histórico (3.6); vínculo (3.9); o filtro `Arquivo possui/não possui` (3.7/3.8). A **capacidade compartilhada de arquivos** (3.7) materializou o substrato **agnóstico de domínio** em `apps/api/src/files/` (`FileObject`/`FileScan` org-scoped RLS+FORCE; `ScanSlot` global): upload→quarentena→verificação composta fail-closed (magic-bytes por conteúdo + 2×SHA anti-troca + ClamAV INSTREAM + if-match na promoção)→promoção atômica a `DISPONIVEL`/`BLOCKED`, download por stream sob sessão, remoção lógica/expurgo. **Zero-dependência**: SigV4 próprio sobre `node:http` (`StorageService`) e clamd INSTREAM sobre `node:net` (`ClamavService`); MinIO/ClamAV só em dev/CI (AD-32). A autz de arquivo **herda do recurso dono** via a porta `FileAuthzContract` (Symbol token; deny-all default), sem `files/` conhecer Card/Registro. A **Story 3.8** ativou o **Campo Arquivo funcional e o anexo geral** consumindo essa capacidade, na **Opção 1** do dono (preserva `Card.valores` append-only — **sem migration, sem GRANT novo**): (a) **Card = anexo geral** — `FileObject(resourceType='CARD', resourceId=cardId)` FORA de `valores` (rotas `cards/:id/files`); (b) **Registro = valor de Campo Arquivo** — referência tipada (`fileId`/`fileId[]`) em `Record.valores` via `editarValores`, com vínculo conferido (DISPONIVEL + `(RECORD, recordId)`); (c) **canal público inline** — `POST /public/forms/:publicId/submit` multipart: o servidor **reserva o `cardId`**, sobe cada arquivo vinculado a `(CARD, cardId)` e **INSERE** o Card já com os `valores` referenciando os `fileId` (INSERT preserva o append-only), com **compensação fail-closed** (sem Card parcial, sem órfão DISPONIVEL) e um **guard de Content-Length** anti-DoS antes do multer. O `FILE` deixou de ser texto em `submission.ts` (referência tipada, modo `rejeitar`/`referencia`; criação de Card/Registro **rejeita** valor FILE — o inline uniforme autenticado é o débito `DEB-3.8-INLINE-UNIFORME`). A autz é roteada por dois **dispatchers** em `apps/api/src/file-authz/` (fora de `files/`): `FileAuthzDispatcher` (CARD→`pipe-authz`, RECORD→`database-authz`; **read-only sob arquivamento** — anexar/remover em recurso arquivado → 409) e `FileEventDispatcher` (porta `FileEventSink`: `FILE_ATTACHED`/`FILE_REMOVED`/`FILE_REPLACED` no `CardHistory`/`RecordHistory`, mesma tx, sem PII), ambos ligados por `FilesModule.register(authz, eventSink)` global. O **gate de consumo** (RF-3): FormVersion publicada com Campo `FILE` + `FILE_UPLOAD_ENABLED=false` → **409 `CAPACIDADE_ARQUIVO_INDISPONIVEL`** (Card e Registro). A coluna FILE é exibível na tabela de Registros (referência, nunca `bucketKey`) e segue gated para filtro/ordenação. **Não existe ainda em 3.8:** read-side do Histórico do Registro (3.6); vínculo Card↔Registro N-N + `correlationId` (3.9); avatar/FR-32 de arquivo (3.10); soft-delete do arquivo anterior na substituição de valor (`DEB-3.8-SOFT-DELETE-ANTERIOR`, a confirmar com o dono — hoje o anterior segue anexo válido).

A autorização FINA por recurso ("config do Pipe" — quem gerencia Fases e monta Formulários; quem revisa submissões públicas; quem opera/concede acesso a um Card) vive **no serviço** via o helper compartilhado `pipe-authz.ts` (DBT-AUTHZ-01), não no guard. `SubmissaoPublica` é org-scoped (RLS); `PublicFormRoute` é a única tabela **global** do domínio (resolução de tenant pré-contexto — AD-10). Nenhuma tem GRANT de DELETE.
- `apps/api/src/databases/` — domínio do **Épico 3** (Stories 3.1–3.5). `DatabasesService` faz **toda** query por `withTenantContext` (sem `where orgId` manual), espelhando o padrão de `pipes/` mas **sem reusar** suas entidades (`Database ≠ Pipe`). `databases.controller.ts` expõe **6 rotas** (`POST /databases` 201; `GET /databases` e `GET /databases/:id`; `PATCH /databases/:id` 200/409; `POST /databases/:id/archive` e `.../restore` 200) — **nenhuma de exclusão** (runtime sem GRANT de DELETE em `Database`). `database-lifecycle.ts` é o núcleo **puro** (arquivar/restaurar idempotentes; gate de renomear-sob-arquivamento). O subdomínio **`grants/`** (3.2 — `database-grants.service`/`database-grants.controller`, rotas sob `POST/GET/PATCH/DELETE /databases/:databaseId/grants`, todas `@Requer('ler','Database')`) concede papéis por Database sobre `DatabaseGrant` (revogar = soft-delete, **sem** GRANT de DELETE). A autorização **fina por recurso** vive no serviço via o helper compartilhado **`database-authz.ts`** (`resolverPoderNoDatabase`/`exigirLerDatabase`/`exigirGerenciarDatabase`/`exigirConcederPapel` — DBT-AUTHZ-01, twin de `pipe-authz.ts`), **sem** tocar o guard/`ability.ts` (C3 congelado). Autorização em 3.1: só o **Admin da Org** (`ler`/`administrar` via CASL). Na **3.2** `ler Database` foi aberto na CASL para a guarda GROSSA (MEMBER/GUEST passam o guard; a autoridade fina decide no serviço, incluindo `listar`/`obter` **não-enumerantes** — RV-1) e a **autoridade hierárquica** de concessão (Admin da Org → qualquer papel; Admin do Database → só MEMBER/VIEWER) mora em `exigirConcederPapel`; `administrar Database` segue Admin-only. Poder diferencial MEMBER×VIEWER = contrato futuro (3.4). O subdomínio **`forms/`** (3.3 — `database-forms.controller` [montagem+evolução] e `database-form-publication.controller` [publicação], rotas sob `databases/:databaseId/form...`, todas `@Requer('ler','Database')`) ativa o **Formulário de Database** **reutilizando o Form Builder canônico** (`FormsService`/`FieldsService`/`FormPublicationService` de `pipes/forms/`, exportados por `PipesModule`, que `DatabasesModule` importa) — **sem segundo builder**; a autorização é roteada por contexto em `pipes/forms/form-authz.ts` (`exigirGerenciarForm`/`resolverPoderNoForm` → `database-authz` quando o alvo é Database). O owner é `Form.databaseId`; o `AlvoFormulario` ganhou `databaseId?` e os localizadores/serviços do builder foram generalizados (aditivo, regressão de E2 verde). O subdomínio **`records/`** (3.4 — `records.service` [criação idempotente + edição de valores], `record-lifecycle.service` [arquivar/restaurar] e o núcleo puro `record-lifecycle.transitions.ts`, expostos por `records.controller` sob `databases/:databaseId/records...`, todas `@Requer('ler','Database')`) materializa **`Record`**/**`RecordHistory`** **reutilizando `submission.ts` (2.7)** para validar contra o snapshot da `FormVersion` publicada do Formulário de Database; a autz fina é `exigirOperarDatabase` (novo em `database-authz`, acorda o MEMBER dormente — VIEWER 403, sem acesso 404). GRANT de `Record` = `SELECT/INSERT` + UPDATE column-scoped (`lifecycleState`/`valores`/`updatedAt`), **sem DELETE**; `RecordHistory` só `SELECT/INSERT` (append-only). `Database ≠ Pipe` e `Card ≠ Registro` preservados. A **3.5** acrescenta a **leitura/tabela** (`records-read.service` + `record-query.core` puro + `records-query.dto`, rota `GET /databases/:databaseId/records` no mesmo controller) — **read-side puro, sem migration/GRANT** (segue `SELECT`): paginação offset + `total` + `colunas`, filtros por tipo e ordenação por Campo sobre `valores` JSONB via **query raw parametrizada sob RLS** (`$transaction([...definirContextoOrg, $queryRaw])`, pois `orderBy` JSON não é nativo no Prisma 6.19.3); allowlist fail-closed no núcleo puro (Campo/operador/valor → 400); `FILE` gated (AD-28); autz por `exigirLerDatabase` (ler ≠ operar; 404 não-enumerante); INV-REPORT-01 por escopo de Database legível. O subdomínio **`records/history/`** (3.6 — `record-history.controller`/`record-history-read.service`/`record-history.dto`, rota `GET /databases/:databaseId/records/:recordId/history`, `@Requer('ler','Database')`) expõe o **Histórico do Registro** — read-side puro sobre `RecordHistory` (append-only desde 3.4), **espelho exato do Histórico do Card (2.17)** no domínio DISTINTO de Registro (DTO/service **locais**, sem reusar Card — `Card ≠ Registro`): cursor determinístico `[createdAt, id]` (teto 100 — NFR-3/4), **projeção allowlist** (só `id`/`type`/`summary`/`actorId`/data-hora — `orgId`/`recordId` e qualquer binário/chave-de-objeto/URL temporária **fora da fronteira**, AD-15/AD-30; a allowlist blinda por construção os eventos de arquivo que 3.8 grava em `RecordHistory`), autz pelo **acesso ATUAL ao Database dono** (`exigirLerDatabase`; o histórico **nunca** concede acesso — 404 não-enumerante, como SC-2105). **Sem migration e sem GRANT novo** (o runtime já tem `SELECT` em `RecordHistory`); a correção é **novo evento** (imutabilidade garantida pelo write-side de 3.4; a 3.6 só lê).
- `apps/api/src/kernel/db/` — `PrismaService` (client do runtime, papel `giraffe_app`) e `tenant-context.ts`. **A conexão é preguiçosa por decisão**: um `$connect()` ansioso mataria o processo com o banco fora no boot, antes de abrir a porta — sem `/health`, sem `/ready`, sem 503.
- `apps/web` — Next.js 16 (App Router, `output: standalone`), React 19, Tailwind 4. `lib/env.ts` lê `API_BASE_URL` (variável de **servidor**, deliberadamente sem `NEXT_PUBLIC_`); `lib/api.ts` consulta `/health` com timeout e devolve **estado honesto e sanitizado** — falha nunca vaza URL interna, stack ou segredo. `GET /healthz` é a liveness da Web: rota local, sem I/O, **não** consulta a API (a saúde do container não pode depender de um serviço terceiro).

**Health vs. readiness são semanticamente distintos**: `GET /health` (liveness, nunca toca o banco) e `GET /ready` (readiness). Desde a Story 1.2, `/ready` **consulta o banco** e devolve **503** quando ele não está apto — a sonda lê uma tabela do schema (`LIMIT 0`), o que prova conexão, migrations aplicadas e GRANT concedido. Um `SELECT 1` provaria só o socket, e um container com o schema ausente responderia `200 ok` para falhar em toda requisição real. Nenhum payload expõe versão, variáveis, paths ou segredos.

### Isolamento multi-tenant (o invariante-mãe)

- Quem isola é o **banco**, não a aplicação: `Organization`, `Membership`, `Pipe`, `PipeGrant`, `Phase`, `Form`, `Field`, `FormVersion`, `Card`, `CardHistory`, `SubmissaoPublica`, `CardGrant`, `CardResponsavel`, `CardPhaseEntry`, `Database`, `DatabaseGrant`, `Record` e `RecordHistory` têm `ENABLE` **e** `FORCE ROW LEVEL SECURITY`. `Account` (AD-10) e `PublicFormRoute` (2.8, resolução de tenant pré-contexto) são **globais e sem RLS** — nenhum pertence a um tenant.
- **Toda tabela organizacional nova replica esse padrão** (`Pipe`, `PipeGrant`, `Phase`, `Form`, `Field`, `Card`, `CardHistory`, `SubmissaoPublica`, `CardGrant`, `CardResponsavel`, `CardPhaseEntry` no Épico 2; `Database`, `DatabaseGrant` no Épico 3): policies `select/insert/update/delete` por `orgId = current_org_id()`, com `WITH CHECK` no INSERT **e** no UPDATE — sem o `WITH CHECK`, um INSERT com `orgId` alheio seria aceito e ficaria invisível, e um UPDATE poderia **mover** a linha para outra Organização.
- **Nenhuma query organizacional pode passar fora de `withTenantContext()`/`withAccountContext()`** (`kernel/db/tenant-context.ts`). O contexto é definido por **transação** (`set_config(..., true)`); com `false` ele grudaria na conexão e vazaria pelo pool.
- **Não existe, e não pode passar a existir, caminho de bypass de RLS alcançável em runtime** (AD-6). O exemplo oficial do Prisma sugere uma `bypass_rls_policy` — proibida aqui.
- **Dois papéis de banco, nunca o mesmo:** `giraffe_app` (runtime; sem `BYPASSRLS`, não é dono das tabelas, DML mínima) e `giraffe_migrator` (dono do schema, só migrations). O processo que atende requisição **nunca** tem em mãos a credencial do dono.
- **O `GRANT` é fronteira de segurança, não detalhe administrativo.** Onde a RLS não alcança, é ele que nega: `Account` não tem policy, então o runtime tem **só `SELECT`** nela — com `DELETE`, a cascata da FK apagaria Memberships de **todas** as Organizações, porque ações referenciais rodam com bypass de row security. Pela mesma razão o runtime não pode criar nem apagar `Organization`. Em `Pipe`, o runtime tem `SELECT/INSERT/UPDATE` e **não** `DELETE`: "sem exclusão definitiva" é garantido pelo banco, não pela ausência de rota — uma rota de DELETE acrescentada por engano amanhã bateria em `permission denied`. Ao conceder um privilégio novo, escreva o teste que prova o escopo dele.
- `$transaction` no client com contexto é **recusada** (erro de compilação e de runtime): a extensão fecha sobre o client raiz, então uma transação externa rodaria em outra conexão, sem contexto. Transação com contexto é escopo da Story 1.3.
- **Mutação de entidade organizacional entra na trilha de auditoria** (`MODELOS_AUDITADOS` em `tenant-context.ts` — hoje `Organization`, `Membership`, `Pipe`, `PipeGrant`, `Phase`, `Form`, `Field`, `FormVersion`, `Card`, `CardHistory`, `SubmissaoPublica`, `PublicFormRoute`, `CardGrant`, `CardResponsavel`, `CardPhaseEntry`, `Database`, `DatabaseGrant`, `Record`, `RecordHistory`), inclusive a **tentativa negada**: um `updateMany` filtrado pela policy volta `{ count: 0 }` com sucesso aparente, e sem isso a tentativa mais óbvia de vandalismo cross-tenant seria registrada como `allowed`. O troco é um falso positivo conhecido: uma operação idempotente legítima (arquivar um Pipe/Fase já arquivado) também produziria `count: 0` — por isso os caminhos idempotentes retornam SEM emitir o `updateMany`.

## Convenções que o código já assume

- **Segredos e configuração:** `.env` nunca é versionado (só `.env.example`); segredos vêm do cofre/ambiente, nunca de imagem, log ou health. Variável obrigatória ausente → falha honesta, sanitizada, listando apenas _nomes_ de variáveis (ver `ConfigValidationError`).
- **Logs:** estruturados (Pino), sempre sanitizados — nunca senhas, tokens, cookies, headers de auth, corpos de e-mail, prompts/respostas de IA ou PII desnecessária.
- **Versões:** fixadas no `pnpm-lock.yaml`. Nunca `latest`; não trocar a stack sem decisão arquitetural registrada. A API de qualquer biblioteca se confere no MCP do Context7 (ver gate acima), não de memória.
- **Migrations:** versionadas e aplicadas como **etapa controlada** (`db:migrate`), nunca no boot do container — um container que migra ao subir transforma cada réplica e cada restart numa tentativa concorrente de DDL. `prisma db push` não substitui migration versionada. O bootstrap dos **papéis** (`apps/api/prisma/bootstrap/00-roles.sql`) é idempotente, roda com papel administrativo e **precede** as migrations: criar papel exige privilégio que nem o migrator tem.
- **Testes:** Vitest com `include: ['test/**/*.test.ts']` — testes ficam em `apps/*/test/`, fora de `src/`. O contrato HTTP (`/health`, `/ready`) e o isolamento por RLS são cobertos por **teste de integração real** (AppModule em porta efêmera; PostgreSQL de verdade): um teste que só chamasse a função de payload continuaria verde se a rota fosse renomeada, e um mock de banco não provaria isolamento nenhum. **No CI, a suíte da API roda em SÉRIE** (`pnpm test:ci` = `vitest run --no-file-parallelism`) — o estado-alvo do isolamento (integração PostgreSQL serial). A flakiness que motivou isso teve a causa raiz encontrada e **corrigida** (TEST-ISO-01, resolvida): (1) `card-access-rls` reusava a conta **global** de ANA num vínculo ACTIVE persistente na Org C, dando a ANA **2 Orgs ativas** → a resolução de contexto sem `x-org-id` caía em **403 transiente** em `pipe-access-http` sob paralelismo — corrigido com conta **dedicada** (`randomUUID`), espelhando `pipe-grants-rls`; (2) o teto default de 5s do Vitest era apertado para os testes HTTP (login real do better-auth ~2s + setup multi-etapa), estourando de forma flaky sob a carga sustentada do run serial — corrigido elevando `testTimeout` (20s) e `hookTimeout` (30s) em `vitest.config.ts`. O cap de `connection_limit` foi **revertido** (a causa nunca foi pool). Localmente, `pnpm test` segue paralelo por padrão; para a suíte cheia use `--no-file-parallelism` (ou `pnpm test:ci`). A regra de ouro: **nunca reusar Ana/Bruno/Carla/Eva do seed em um `membership.create` que persista** — são fixtures de **LEITURA**; escreva na **Org C** com conta descartável, ou o seu teste quebra o do vizinho por contagem/contexto. A dívida **TEST-ISO-01** segue aberta apenas para o aceite formal (≥20 runs paralelos verdes) antes de restaurar o paralelismo pleno por grupos. O typecheck da API cobre `src` **e** `test` (`tsconfig.json`); o build usa `tsconfig.build.json`, que exclui os testes de `dist`.
- **Um teste pode passar pelo motivo errado.** Nesta base isso já aconteceu duas vezes: o `create` do Prisma emite `INSERT ... RETURNING`, e o RETURNING esbarra na policy de **SELECT** — o teste de inserção cruzada ficava verde mesmo com o `WITH CHECK` desligado (por isso existe a versão com `createMany`, que não tem RETURNING). E o teste de "não é dono das tabelas" nunca olhava para `relowner`. Ao escrever um teste de segurança, **prove a fase vermelha**: quebre a policy de propósito e confirme que ele falha.
- **ESLint ignora** `docs/`, `_bmad/`, `_bmad-output/`, `skills/`, `specs/`, `.specify/`, `.github/`, `.agent*/` — são tooling e documentação, não código de aplicação.
- **CI:** `.github/workflows/ci.yml` — 4 jobs (`qualidade`, `testes`, `containers`, `seguranca`). O banco do CI sobe pelo **Docker Compose**, para que exista **uma única** definição de provisionamento de papéis; reescrevê-la no YAML criaria uma segunda verdade, e a que vale em produção seria a que ninguém testa. Senhas de CI são geradas por execução; actions são fixadas por **SHA**, nunca por tag (tag é mutável).
- **Testes verdes ≠ afirmação:** um critério de aceite só é marcado como concluído com evidência de execução real (Constitution X).

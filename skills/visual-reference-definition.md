# Skill — Referências Visuais do Giraffe CRM

## 1. Objetivo

Transformar referências visuais em uma direção própria para o Giraffe CRM.

A Skill deve responder:

**“Como o Giraffe CRM deve parecer, se comportar visualmente e manter consistência entre Formulários, Cards, Pipes, Databases, Automações, Conversas, Arquivos como capacidade contextual, IA, Integrações e Dashboard?”**

A direção visual deve:

- ser limpa;
- ser profissional;
- ser moderna;
- transmitir confiança;
- suportar alta produtividade;
- manter densidade equilibrada;
- evitar aparência genérica de template;
- evitar aparência exagerada de IA;
- usar o laranja `#FF7200` com moderação;
- preservar consistência entre todas as áreas.

---

## 2. Entradas obrigatórias

Antes de criar ou alterar a direção visual, consulte:

- Visão do Produto aprovada;
- MVP aprovado;
- Regras de Negócio aprovadas;
- Permissões aprovadas;
- Modelagem de Dados aprovada;
- Mapeamento de Integrações Externas aprovado;
- referências visuais enviadas;
- logotipo e identidade da marca, quando existirem;
- decisões visuais já aprovadas.

Para o Giraffe CRM, considere obrigatoriamente:

- público inicial: agências de marketing;
- o produto é um SaaS operacional;
- o usuário trabalha com muitos clientes, cards, arquivos e conversas;
- Formulário, Database e Pipe possuem responsabilidades diferentes;
- o Form Builder deve ser configurável por serviço;
- cada Fase pode possuir tarefas próprias;
- cada Fase pode possuir responsáveis próprios;
- o Card deve mostrar o estado atual e o Histórico anterior;
- Automações seguem o modelo Quando → Condições → Então;
- Automações devem mostrar estado, última execução e resultado;
- Logs detalhados e Histórico operacional possuem finalidades diferentes;
- IA pode aparecer como ação possível dentro de Automação;
- Pipe e Database precisam parecer relacionados, mas não iguais;
- Dashboard deve mostrar o que exige atenção;
- IA deve parecer assistiva, não autônoma;
- integrações precisam mostrar estados e falhas;
- a aplicação web deve funcionar bem em celular;
- a interface deve ser simples na superfície, mesmo quando o produto for poderoso por baixo.

---

## 3. Fonte de verdade e conflitos visuais

Não substitua silenciosamente uma decisão visual aprovada.

Ao encontrar conflito entre:

- direção visual;
- referência externa;
- recomendação da UI UX Pro Max;
- Design System;
- tela existente;

faça:

1. identifique a decisão aprovada;
2. descreva a contradição;
3. não substitua automaticamente;
4. registre em **Conflitos e decisões pendentes**;
5. proponha alternativas compatíveis;
6. atualize a fonte oficial somente após aprovação explícita.

### Regra

Referências e ferramentas externas orientam.

A direção visual aprovada do Giraffe CRM decide.

---

# 4. Norte visual

A frase central da identidade visual é:

**“Muito poder por baixo, pouca distração por cima.”**

A interface deve transmitir:

- clareza;
- controle;
- organização;
- agilidade;
- confiança;
- consistência.

A interface não deve depender de:

- gradientes decorativos;
- sombras fortes;
- excesso de cores;
- animações chamativas;
- cards coloridos sem função;
- elementos “futuristas” apenas para representar IA.

---

# 5. Personalidade visual

## Deve parecer

- clean;
- profissional;
- moderno;
- direto;
- organizado;
- confiável;
- produtivo;
- leve sem ser vazio;
- amigável sem ser infantil.

## Não deve parecer

- ERP antigo;
- dashboard genérico de template;
- interface de jogo;
- produto infantil;
- sistema excessivamente colorido;
- painel cheio de gradientes;
- “IA futurista” com neon;
- cópia literal de outro produto.

---

# 6. Referências aprovadas e papel de cada uma

## 6.1 Referência principal — sistema visual neutro e editorial

A referência principal enviada pelo projeto contribui com:

### Aproveitar

- interface quase toda neutra;
- branco, preto e cinzas como base;
- poucas sombras;
- bordas discretas;
- densidade alta sem sensação de bagunça;
- 8px como ritmo-base de espaçamento;
- forte consistência entre telas;
- cor reservada para momentos importantes;
- superfícies planas;
- hierarquia tipográfica clara.

### Não copiar

- identidade da marca original;
- fontes proprietárias;
- uso excessivo de pills;
- estrutura de e-commerce;
- hero editorial gigante;
- componentes específicos de varejo;
- preto como CTA principal universal.

### Adaptação para o Giraffe CRM

Substituir o CTA preto dominante por:

**Laranja Giraffe `#FF7200` usado seletivamente.**

---

## 6.2 Airbnb — suavidade e clareza

### Aproveitar

- sensação limpa;
- superfícies claras;
- hierarquia simples;
- interface amigável;
- boa legibilidade.

### Não copiar

- espaçamento excessivo para um CRM;
- estrutura de marketplace;
- dependência de fotografia;
- pills em excesso.

---

## 6.3 Linear — clareza e produtividade

### Aproveitar

- hierarquia;
- densidade;
- foco;
- estados bem definidos;
- sensação de produto rápido.

### Não copiar

- identidade escura como padrão;
- excesso de minimalismo que esconda ações;
- linguagem visual proprietária.

---

## 6.4 Airtable / Attio — dados e relacionamentos

### Aproveitar

- visualização de informação estruturada;
- tabelas densas;
- relacionamento entre registros;
- painéis laterais;
- edição contextual.

### Não copiar

- estrutura exata;
- identidade;
- terminologia proprietária;
- comportamento que conflite com Pipe e Database do Giraffe CRM.

---

## 6.5 UI UX Pro Max — especialista de apoio

A UI UX Pro Max pode ser usada para:

- pesquisar padrões;
- comparar alternativas;
- revisar UX;
- sugerir componentes;
- verificar responsividade;
- identificar anti-patterns;
- apoiar a criação do Design System.

Ela não é a fonte de verdade da identidade visual.

### Regra

Toda recomendação da UI UX Pro Max deve:

1. respeitar a Visão do Produto;
2. respeitar esta Skill;
3. respeitar decisões aprovadas;
4. preservar consistência entre áreas;
5. ser rejeitada quando contradizer a direção oficial.

# 7. Princípio de uso de cor

A interface deve seguir aproximadamente:

**85–90% neutros + 10–15% cor de identidade e estados.**

O laranja deve orientar.

Não deve dominar.

### Regra

Se muitos elementos laranja aparecem ao mesmo tempo, nenhum deles parece prioritário.

# 9. Cores semânticas

O laranja da marca não deve representar todos os estados.

## Erro

`#D92D20`

Uso:

- erro;
- falha;
- ação destrutiva.

## Sucesso

`#157A52`

Uso:

- concluído;
- integração saudável;
- ação confirmada.

## Informação

`#2563EB`

Uso:

- informação;
- estado informativo.

## Atenção

`#A15C00`

Uso:

- risco;
- atraso;
- atenção operacional.

### Regra

Não usar `#FF7200` para:

- erro;
- sucesso;
- alerta;
- indisponibilidade;
- falha de integração.

O laranja representa identidade e ação.

Os estados semânticos representam significado.

# 11. Sistema de espaçamento

## Unidade base

8px

## Escala conceitual

- 4px — micro ajuste;
- 8px — relação direta;
- 12px — componente compacto;
- 16px — componente padrão;
- 24px — bloco;
- 32px — seção interna;
- 48px — grande separação.

### Regra

Espaço deve criar organização.

Não usar espaço vazio apenas para parecer premium.

---

# 12. Densidade

A densidade padrão do Giraffe CRM deve ser:

**equilibrada para alta produtividade.**

O produto precisa acomodar:

- clientes;
- cards;
- fases;
- arquivos relacionados;
- conversas;
- follow-ups;
- indicadores.

### Regra

A interface deve mostrar informação suficiente sem obrigar o usuário a abrir várias telas para entender o estado da operação.

### Evitar

- excesso de padding;
- cards gigantes;
- tabelas com linhas altas demais;
- espaço desperdiçado;
- excesso de informação ao mesmo tempo.

---

## Botões

Radius:

8px

### Regra

Botões não devem ser pills por padrão.

Pills ficam reservadas para:

- chips;
- filtros;
- status;
- tags.

---

## Inputs

Radius:

8px

---

## Cards

Radius:

10–12px

### Tratamento

Preferir:

- superfície;
- borda leve;
- separação por espaço.

Evitar:

- sombra pesada;
- elevação exagerada.

---

## Modais e painéis

Radius:

12–16px

---

## Avatares

Circular.

# 14. Elevação e profundidade

A interface deve ser predominantemente plana.

## Nível 0

Sem sombra.

Uso:

- cards;
- tabelas;
- seções.

## Nível 1

Borda de 1px.

Uso:

- separação;
- card selecionável;
- container.

## Nível 2

Sombra muito leve.

Uso somente quando necessário para:

- dropdown;
- popover;
- modal;
- elemento flutuante.

### Regra

Não usar sombra para tentar tornar cada card “premium”.

---

# 15. Hierarquia de botões

## 15.1 Botão primário

### Visual

- fundo `#FF7200`;
- texto `#111111`;
- radius 8px;
- peso 600.

### Uso

- criar;
- salvar;
- continuar;
- enviar;
- aprovar;
- conectar;
- concluir ação prioritária.

### Regra

Preferir uma ação primária clara por região visual.

---

## 15.2 Botão secundário

### Visual

- fundo branco ou `#F5F5F5`;
- texto `#111111`;
- borda `#E5E5E5`.

### Uso

- editar;
- cancelar;
- filtrar;
- ações de apoio.

---

## 15.3 Botão terciário

### Visual

- sem fundo;
- texto `#39393B`.

### Uso

- ver detalhes;
- abrir;
- ações de menor prioridade.

---

## 15.4 Botão destrutivo

### Visual

Usar vermelho semântico.

### Regra

Nunca usar laranja para exclusão destrutiva.

---

## 15.5 Botão de ícone

Pode ser:

- circular;
- 40–44px;
- fundo neutro;
- estado ativo laranja suave.

# 16. Uso moderado do laranja

## Usar laranja em

- CTA principal;
- botão principal;
- navegação ativa;
- tab ativa;
- foco;
- seleção;
- ação humana de aprovação;
- pequeno indicador de identidade;
- estado ativo de controle.

## Usar laranja suave em

- linha selecionada;
- conversa selecionada;
- item ativo;
- fundo de navegação ativa;
- destaque leve de sugestão.

## Não usar laranja em

- todos os cards;
- todos os ícones;
- sidebar inteira;
- header inteiro;
- fundo principal;
- gráficos inteiros;
- todos os botões;
- todos os badges.

### Regra

Antes de usar laranja, perguntar:

**“Este elemento realmente precisa ser percebido como ação, seleção ou identidade?”**

Se não, usar neutro.

---

# 17. Navegação principal

## Sidebar

Base:

- branca;
- ou cinza muito claro.

### Item normal

- texto escuro;
- ícone neutro.

### Item ativo

Preferir uma destas opções:

#### Opção A

- fundo `#FFF3E8`;
- texto `#111111`;
- ícone laranja.

#### Opção B

- fundo neutro;
- pequena barra lateral `#FF7200`.

### Regra

Não usar sidebar inteira laranja.

---

## Header

Preferir:

- branco;
- borda inferior leve;
- ações organizadas;
- CTA primário laranja somente quando necessário.

---

# 18. Dashboard

A pergunta central do Dashboard é:

**“O que precisa da minha atenção agora?”**

## Ordem visual

1. atrasados;
2. follow-ups;
3. itens sem atividade;
4. operação em andamento;
5. visão geral.

## Uso do laranja

Usar em:

- CTA principal;
- filtro ativo;
- seleção;
- pequena ênfase de identidade.

Não usar laranja para todos os números.

### Estados operacionais

Usar cores semânticas.

Exemplo:

- atraso → atenção;
- falha → erro;
- concluído → sucesso;
- ação principal → laranja.

---

# 19. Pipe

## Direção visual

- cards neutros;
- fases claras;
- alta legibilidade;
- informação principal visível;
- pouca decoração.

## Card deve priorizar

- identificação;
- responsável;
- prazo;
- estado;
- alertas.

## Uso do laranja

- card selecionado;
- foco;
- botão criar card;
- ação principal;
- indicador ativo.

## Não usar

- todos os cards com fundo laranja;
- cada fase com uma cor forte diferente.

---

# 20. Database

## Direção visual

O Database deve parecer:

- estruturado;
- informacional;
- produtivo;
- diferente de um Kanban.

## Priorizar

- tabela;
- lista;
- painel lateral;
- relacionamentos;
- filtros;
- campos.

## Uso do laranja

- célula ativa;
- linha selecionada;
- filtro ativo;
- botão criar registro;
- foco.

## Regra

O laranja não deve reduzir a legibilidade de grandes volumes de dados.

# 20A. Form Builder

## Direção visual

O Form Builder deve parecer:

- configurável;
- claro;
- direto;
- produtivo;
- orientado a montagem.

A estrutura preferencial é:

```text
┌────────────────┬───────────────────────────┬──────────────────┐
│ CAMPOS         │ FORMULÁRIO                │ CONFIGURAÇÃO     │
│                │                           │                  │
│ Texto curto    │ Dados da empresa          │ Nome do campo    │
│ Texto longo    │                           │ Obrigatório      │
│ E-mail         │ [____________________]    │ Placeholder      │
│ Telefone       │                           │ Ajuda            │
│ Número         │ Materiais                 │                  │
│ Data           │ [ Enviar arquivos ]       │ DESTINO          │
│ Seleção        │                           │ Database         │
│ Arquivo        │                           │ Pipe             │
└────────────────┴───────────────────────────┴──────────────────┘
```

### Esquerda

- biblioteca de campos.

### Centro

- formulário em construção.

### Direita

- configuração do elemento selecionado.

### Regra

Preferir área completa de trabalho.

Evitar modal gigante e apertado.

---

## Biblioteca de campos

Deve permitir reconhecer rapidamente:

- texto curto;
- texto longo;
- e-mail;
- telefone;
- número;
- data;
- seleção;
- checkbox;
- Arquivo;
- seção;
- texto informativo.

### Visual

Preferir:

- lista compacta;
- ícones neutros;
- rótulos claros;
- hover discreto;
- seleção em laranja suave.

---

## Canvas do Formulário

Deve permitir:

- leitura da estrutura;
- reordenação;
- seleção do campo;
- visualização de seções;
- estado vazio útil.

### Regra

O Canvas deve parecer Formulário.

Não deve parecer Database nem Kanban.

---

## Painel de configuração

Deve mudar conforme o elemento selecionado.

Pode exibir:

- nome;
- orientação;
- obrigatoriedade;
- placeholder;
- opções;
- Destino dos Dados.

### Regra

O painel contextual deve reduzir a quantidade de modais.

---

## Destino dos Dados

O usuário precisa entender visualmente:

```text
Formulário
    ↓
Database / Registro
    ↓
Pipe / Card
```

### Direção

Usar:

- labels claras;
- seleção por recurso;
- resumo do fluxo;
- linguagem de negócio.

Evitar:

- payloads;
- schemas;
- aparência de configuração técnica.

---

## Ações principais

Priorizar:

- Pré-visualizar;
- Salvar;
- Publicar.

### Uso do laranja

`Publicar` pode ser ação primária quando o Formulário estiver pronto.

---

# 20B. Visualização detalhada do Card

## Objetivo

Permitir compreender:

- onde o processo está;
- o que precisa acontecer agora;
- quem é responsável;
- o que já aconteceu antes.

## Estrutura preferencial

```text
┌──────────────────┬──────────────────────────┬──────────────────┐
│ CONTEXTO         │ EXECUÇÃO ATUAL           │ AÇÕES            │
│                  │                          │                  │
│ Informações      │ Fase atual               │ Mover Card       │
│ Arquivos         │                          │ Alterar resp.    │
│                  │ Tarefas                  │ Outras ações     │
│ Histórico        │ Campos da etapa          │                  │
└──────────────────┴──────────────────────────┴──────────────────┘
```

### Esquerda

- informações gerais;
- Arquivos relevantes;
- Histórico anterior.

### Centro

- Fase atual;
- tarefas atuais;
- responsáveis;
- campos e dados da etapa.

### Direita

- mudança de Fase;
- ações rápidas;
- ações complementares.

### Regra de equilíbrio

O estado atual deve ter maior prioridade visual.

O Histórico deve permanecer visível ou facilmente acessível.

---

# 20C. Tarefas e responsáveis por Fase

## Tarefas da Fase

As tarefas devem parecer ligadas à Fase atual.

Exemplo:

```text
FASE ATUAL
Produção

TAREFAS

○ Criar arte
✓ Revisar conteúdo
○ Preparar versão final
```

A interface deve permitir distinguir:

- pendente;
- em andamento quando aplicável;
- concluída;
- reaberta quando aplicável.

### Regra

Não depender apenas de cor.

Usar:

- ícone;
- label;
- texto;
- posição.

---

## Execução real

Uma tarefa concluída pode mostrar:

```text
✓ Revisar briefing

Concluída por Martins
Hoje às 09:42
```

### Regra

A interface deve deixar claro:

```text
Tarefa configurada na Fase
        ≠
Execução real no Card
```

---

## Responsáveis por Fase

A interface deve diferenciar:

```text
Responsável padrão da Fase
```

de:

```text
Responsável atual do Card
```

Exemplo:

```text
FASE: PRODUÇÃO

Equipe padrão
Design

RESPONSÁVEL ATUAL

Martins Junior
```

### Regra

Não mostrar os dois conceitos como se fossem o mesmo dado.

Pode usar:

- avatar;
- nome;
- equipe;
- label contextual.

---

# 20D. Histórico do Card

## Direção visual

O Histórico deve permitir leitura cronológica.

Exemplo:

```text
HISTÓRICO

● Card criado
  por Martins
  09:02

● Entrou em Produção
  por Junior
  09:15

● Tarefa concluída
  Revisar briefing
  09:42

● Arquivo adicionado
  identidade-visual.pdf
  10:03

● Automação executada
  Criar Card de Artes
  10:05
```

## Hierarquia

Cada evento deve destacar:

- ação principal;
- ator ou origem;
- momento;
- contexto relevante.

### Regra

Não transformar cada evento em Card grande.

Preferir:

- timeline;
- lista compacta;
- agrupamento cronológico.

## Tipos de evento

Podem incluir:

- criação;
- mudança de Fase;
- tarefa;
- responsável;
- comentário;
- Arquivo;
- Automação;
- IA;
- ação externa.

### Regra

O Histórico deve ser visualmente consistente mesmo com tipos diferentes de evento.

### Histórico e Log

O Histórico pode mostrar:

```text
“Automação executada”
```

O Log detalhado pode mostrar:

- gatilho;
- condições;
- ação;
- tentativas;
- resultado.

### Regra

Não despejar Log técnico no Histórico do Card.

---

# 20E. Automações

## Direção geral

O módulo deve ser simples na superfície.

O usuário deve entender:

```text
QUANDO
        ↓
CONDIÇÕES
        ↓
ENTÃO
```

## Tela de lista

Estrutura recomendada:

```text
Automações | Logs

┌──────────────────────────────────────────────────────┐
│ Follow-up de cliente parado               Ativa  ⋮  │
│                                                      │
│ QUANDO                     ENTÃO                     │
│ Card sem atividade    →    Pedir sugestão à IA      │
│                                                      │
│ Última execução: Hoje 09:32                          │
│ Resultado: Concluído                                 │
└──────────────────────────────────────────────────────┘
```

A lista deve permitir perceber:

- nome;
- gatilho;
- ação;
- estado;
- última execução;
- resultado recente;
- responsável pela última alteração quando aplicável.

## Editor de Automação

Estrutura preferencial:

```text
QUANDO

┌───────────────────────────┐
│ Card ficar sem atividade  │
└─────────────┬─────────────┘
              ↓

CONDIÇÕES

┌───────────────────────────┐
│ Processo ainda está ativo │
└─────────────┬─────────────┘
              ↓

ENTÃO

┌───────────────────────────┐
│ Pedir sugestão à IA       │
└───────────────────────────┘
```

### Regra

A conexão visual deve ajudar a leitura.

Não deve virar diagrama complexo no MVP.

## Uso do laranja

Usar em:

- botão Nova Automação;
- ação principal;
- seleção;
- conector entre blocos;
- foco.

Não usar em:

- todos os blocos;
- todos os estados;
- falhas.

## Estados da Automação

A interface deve distinguir, no mínimo:

- ativa;
- pausada;
- com problema quando aplicável.

### Regra

O estado deve usar:

- texto;
- ícone ou label;
- cor semântica quando necessário.

---

# 20F. Logs e resultados de Automação

## Objetivo

Permitir responder:

- qual Automação executou;
- por que executou;
- o que tentou fazer;
- qual foi o resultado.

## Estados mínimos de resultado

A interface deve distinguir:

- concluído;
- não concluído;
- aguardando resultado;
- precisa de atenção.

### Regra

`Aguardando resultado` não deve parecer sucesso.

## Visual do Log

Exemplo:

```text
AUTOMAÇÃO
Follow-up de cliente parado

GATILHO
Card ficou sem atividade

CONDIÇÕES
Processo ativo: Sim

AÇÃO
Solicitar sugestão à IA

RESULTADO
Concluído
```

Pode incluir timeline interna quando necessário.

## Densidade

O Log deve parecer:

- operacional;
- legível;
- rastreável.

Não deve parecer:

- console técnico;
- terminal;
- dump de payload.

Detalhes técnicos pertencem a níveis apropriados de observabilidade.

---

# 21. Conversas

## Direção visual

A área de Conversas deve deixar claro:

- quem falou;
- em qual canal;
- quando;
- se exige resposta;
- se existe follow-up;
- se é mensagem externa ou nota interna.

## Uso do laranja

- conversa selecionada com fundo suave;
- ação principal de envio;
- aprovação humana de follow-up;
- foco.

## Não usar

- balões inteiros laranja por padrão;
- nota interna com o mesmo visual da mensagem externa.

---

# 22. IA

A IA deve parecer:

- assistiva;
- clara;
- revisável;
- controlada.

Ela não deve parecer:

- autônoma;
- mágica;
- misteriosa;
- decorada com neon.

## Estados visuais obrigatórios

- sugestão gerada;
- aguardando revisão;
- editada pelo humano;
- aprovada;
- descartada;
- ação executada.

## Uso do laranja

Preferir:

- pequeno ícone;
- linha lateral;
- label;
- fundo suave;
- botão final de aprovação.

### Regra

Não transformar toda área da IA em laranja.

A cor deve reforçar a decisão humana.

---

# 23. Arquivos como capacidade contextual

## Direção visual

Arquivos não são uma área visual principal independente no MVP.

Eles aparecem dentro dos contextos aos quais pertencem:

- Database / Registro;
- Card;
- Formulário;
- Conversa / Mensagem.

Preservar:

- upload;
- download;
- origem;
- relacionamento;
- permanência após conclusão do Card quando o arquivo for importante para o contexto persistente.

No Database / Registro, pode mostrar:

- nome;
- tipo;
- tamanho;
- origem;
- data;
- quem enviou;
- upload;
- download.

No Card, pode mostrar:

- arquivos relevantes ao processo;
- origem;
- relação com o Registro;
- upload;
- download.

No Formulário, pode existir campo de upload.

Fluxo:

```text
Formulário
    ↓
Submissão
    ↓
Arquivo
    ↓
Registro relacionado
```

Em Conversas, anexos podem aparecer relacionados à Mensagem e, quando necessário, também ao contexto persistente apropriado.

## Uso do laranja

- upload principal;
- item selecionado;
- ação prioritária.

## Não usar

- File Manager independente no MVP;
- biblioteca global de arquivos;
- Google Drive interno;
- gerenciador de pastas;
- área principal chamada "Files";
- miniaturas ou cards coloridos sem função.

### Regra

A mudança é visual e organizacional.

Não remova a entidade Arquivo do produto.

Não decida armazenamento técnico nesta Skill.

---

# 24. Integrações

A interface deve mostrar claramente:

- ativa;
- desconectada;
- atenção necessária;
- ação pendente;
- resultado desconhecido;
- falha.

## Regra

O laranja não deve representar sozinho a saúde da integração.

Usar estados semânticos.

### Uso do laranja

- conectar integração;
- reautorizar como ação principal;
- item selecionado.

---

# 25. Estados do sistema

A direção visual deve definir padrões para:

- loading;
- vazio;
- erro;
- sem permissão;
- desconectado;
- integração com problema;
- ação pendente;
- resultado desconhecido;
- IA processando;
- IA aguardando revisão;
- item atrasado;
- item sem atividade.

### Regra

Estados não podem depender apenas de cor.

Usar também:

- texto;
- ícone;
- label;
- estrutura.

# 26. Responsividade

O Giraffe CRM é web responsivo.

Não é necessário aplicativo nativo no MVP.

## No celular, deve ser possível

- consultar cliente;
- abrir Card;
- mover Card;
- visualizar tarefas da Fase;
- concluir tarefa;
- visualizar responsáveis;
- visualizar Histórico essencial;
- visualizar arquivos no contexto do Registro ou Card;
- acompanhar Conversa;
- verificar pendências;
- revisar Sugestão de IA;
- aprovar Follow-up;
- preencher Formulário público;
- consultar resultado essencial de Automação.

## Pode permanecer prioritariamente desktop

- configurar Pipe;
- configurar Database;
- configurar Form Builder;
- criar ou editar Automação;
- gerenciar Permissões;
- configurar Integrações;
- administração complexa.

# 27. Breakpoints conceituais

A Skill não deve fixar framework.

Use os intervalos apenas como direção:

## Desktop amplo

1440px+

- maior densidade;
- mais colunas;
- painéis simultâneos.

## Desktop

1024–1439px

- layout padrão.

## Tablet

768–1023px

- redução de colunas;
- painéis recolhíveis.

## Mobile

320–767px

- uma coluna;
- ações principais acessíveis;
- navegação adaptada;
- sem esconder contexto essencial.

# 28. Toques e acessibilidade

Todo controle principal deve possuir área de toque adequada.

Direção mínima:

- 44px para ações principais em mobile;
- estados de foco visíveis;
- contraste suficiente;
- informação não dependente apenas de cor.

### Regra

Acessibilidade não deve ser tratada como ajuste final.

Ela faz parte da direção visual.

# 29. Do

- usar laranja apenas onde há prioridade;
- manter texto quase preto;

- não usar laranja como sucesso;

# 36. Princípio final

Sempre prefira:

**base neutra + hierarquia forte + densidade equilibrada + laranja estratégico**

em vez de:

**interface colorida em que tudo compete por atenção.**

As perguntas finais são:

**“Se retirarmos o laranja de todos os elementos não essenciais, a interface continua clara e o que sobra em laranja realmente indica prioridade?”**

**“O usuário consegue entender o que precisa fazer agora e o que já aconteceu antes?”**

**“Uma Automação parece simples de compreender sem esconder seu resultado?”**

**“Formulário, Database e Pipe parecem partes relacionadas do mesmo produto, mas com funções visuais distintas?”**

Se as respostas forem sim, a direção visual está preservando a identidade e a usabilidade do Giraffe CRM.

01Resumo executivo

O Giraffe CRM está maduro como referência visual e de interação. O núcleo — o pipe Kanban — é a peça mais completa do sistema: entrega 7 visualizações (Fluxo, Kanban, Lista, Relatórios, Formulário, E-mails, Painéis), modal de card com comentários, e-mail, checklist, automações e mover fases, além de configurações completas (formulário, fases, pessoas, e-mail, config do pipe), conexões, formulário público e convites. Database, Automações e Agentes de IA seguem o mesmo vocabulário visual e reaproveitam motores compartilhados (field-config.js, email-templates.js).

O que impede tratar o conjunto como "produto coerente" hoje não é o visual — é a ausência de um modelo de dados documentado e uma série de inconsistências de dados/identidade entre telas. Os protótipos foram construídos isoladamente: cada tela inventa seu próprio catálogo de pipes, sua própria identidade de usuário e sua própria fonte de notificações. Não há documentação de regras de negócio, glossário ou matriz de permissões — apenas HTML. Para a Fase 1 avançar para implementação, a prioridade é consolidar o modelo conceitual, não construir mais telas.

Veredito
Pronto como referência de UI · Não pronto como especificação de produto. Nenhum bloqueador visual. Os bloqueadores são conceituais: modelo de dados, identidade/multi-tenant e regras de negócio ainda vivem só na cabeça de quem fez os protótipos.

03Bloqueadores 🔴

Impedem a passagem de "protótipo" para "especificação implementável". Não são bugs de tela.

B1 · Não existe modelo de dados documentado

Toda a estrutura conceitual (o que é um Pipe, uma Fase, um Card, um Campo, um Registro, uma Automação, um Agente) só existe implícita no HTML. Não há um documento que defina entidades, atributos, relacionamentos e cardinalidade.

Evidência: a pasta docs/01-documentacao-base/ contém apenas 07-referencias-visuais/prototypes/. Não há seções 01–06, glossário, nem arquivo de regras. As seções 01–06 estão numeradas mas ausentes.

B2 · Identidade de usuário vs. empresa é contraditória

O sistema não deixa claro se "Giraffe Marketing" é o usuário, a empresa/organização, ou ambos — e o nome exibido muda entre telas.

Evidência: em dashboard-home.html o canto superior direito e a saudação mostram "Martins junior / MJ"; em todas as outras telas o mesmo canto mostra "Giraffe Marketing / GM". O menu de usuário lista "Giraffe Marketing" como nome + "Super Administrador" como se fosse pessoa, mas o mesmo texto aparece como organização de cobrança no token. O campo "Nome" do perfil = "Giraffe Marketing", e-mail = martinsjunior089@gmail.com.

B3 · Multi-tenant é prometido mas não existe

"Trocar de empresa" aparece no menu de usuário de toda tela, e o token tem "Organização de cobrança" com múltiplas empresas — mas não há fluxo de troca de empresa nem definição de como pipes/databases se isolam por organização.

Evidência: botão "Trocar de empresa" é inerte em dashboard-home, tarefas-solicitacoes, meu-perfil, relatorios-empresa, minhas-notificacoes. Nenhum seletor de empresa em lugar algum.

04Problemas importantes 🟡

P1 · Catálogo de pipes diferente em cada tela

Não há uma lista canônica de pipes. Cada tela inventa a sua.

  • Início: só "Contratos e Juridicos" é real; os outros 4 (Criação de Artes, Site e Manutenção, Estrutura de Implementação, Estrutura Tráfego 6666) estão marcados "Indisponível neste protótipo".
  • Relatórios da empresa: botão diz "13 pipes", mas a lista tem só 10 — e com nomes que não aparecem no Início: "[SO] Solicitação de Orçamento", "[TP] Tráfego - Lucas/Hiago/Matheus", "Estrutura do Tráfego".
  • Perfil → Notificações: aparece "Contratos e Juridicos (cópia 1)", que não existe em nenhuma outra tela.
  • Automações: o seletor "No pipe" repete a lista de 10 pipes dos relatórios.

Impacto: impossível saber quantos pipes o sistema tem, ou quais nomes são canônicos.

P2 · Notificações têm duas fontes de dados divergentes

O popover global (em toda tela) tem 7 itens fixos com datas absolutas (07/07/2026 13:48). A página completa tem 15 itens com tempo relativo ("há 7 minutos"). O badge diz sempre "9+". Os três números não conversam.

P3 · "Tudo em dia" contradiz o resto do sistema

tarefas-solicitacoes.html mostra "Sem tarefas / Tudo em dia!" e "Nenhuma solicitação" — enquanto as notificações listam vários cards "atrasado", "expirado" e "vencido". Um usuário com 5 cards atrasados não deveria ver "Tudo em dia".

P4 · Relatórios exibe dados obviamente sintéticos

Em relatorios-empresa.html, todas as ~27 linhas têm Fase = "Arquivado" e criador = "Jhenipher martins". O cabeçalho anuncia "3769 resultados / 1–50", mas só 27 linhas são renderizadas e o rodapé diz "1 – 50 de 3769". Números internamente inconsistentes.

P5 · Vazamentos de escopo da Fase 2 sem gate

Alguns recursos de Fase 2 (API externa, webhooks) aparecem ativos, sem a marca "Em breve" que o resto do sistema usa corretamente:

  • Token de acesso pessoal (Perfil) referencia a GraphQL API — API externa é Fase 2.
  • Automações → "Faça uma requisição HTTP" entrega um editor de corpo HTTP completo — webhook/HTTP é Fase 2.
  • Painel → API está corretamente marcado "Em breve" e bloqueado — este é o padrão certo. E as "Chaves de API" com permissão "Webhooks" no painel também são Fase 2.

Nota: a criação de "card/registro conectado" em Automações está corretamente marcada "Ação em desenvolvimento". O gating é inconsistente: alguns recursos de Fase 2 são gateados, outros não.

06Funcionalidades faltantes (dentro da Fase 1)

Recursos que a Fase 1 pressupõe mas que nenhum protótipo cobre.

  • Busca global funcional — prometida pela UI em toda tela, sem nenhuma implementação de resultados.
  • Estado preenchido de Tarefas e Solicitações — só existe o estado vazio; não há exemplo de tarefa/solicitação real, nem o fluxo de abrir/concluir.
  • Fluxo de criação real de pipe/database — o modal "Criar pipe/database" só fecha; não leva a um pipe novo nem ao "Criar com IA".
  • Múltiplos pipes e databases navegáveis — apenas 1 pipe e 1 database abrem; todo o resto é placeholder. A troca entre pipes não é demonstrável.
  • Envio real de e-mail / convite — os fluxos chegam ao estado de sucesso, mas não há caixa "enviados" nem histórico consolidado.
  • Datepicker para vencimento de card e campos de data de formulário.
  • Estado logado ↔ deslogado — "Sair" não retorna ao login; não há guarda de sessão.

08Inconsistências conceituais

  • Usuário ≠ Empresa ≠ Marca. "Martins junior" (pessoa), "Giraffe Marketing" (marca/empresa) e o papel "Super Administrador" são usados de forma intercambiável. Ver B2.
  • Tarefa vs. Solicitação vs. Card. Três entidades coexistem sem relação definida. As notificações falam de "cards"; a tela de trabalho fala de "tarefas" e "solicitações"; os relatórios listam "cards" como linhas.
  • Pipe vs. Database. Ambos têm "Formulário inicial", "Pessoas", "Configurações", ícone e exclusão — vocabulário quase idêntico. Falta documentar em que diferem conceitualmente (fluxo/fases vs. tabela de registros).
  • "Assistentes de IA" em dois lugares. Existe um item global no menu de usuário e uma tela de Agentes por-pipe. Não está claro se IA é global, por-pipe, ou os dois.
  • Notificação vs. Log vs. Auditoria. Há "Notificações", "Logs" (em Automações e Agentes) e "Auditoria" (no Painel). Três históricos com fronteiras não definidas.

09Inconsistências visuais

O sistema de design é forte e consistente na maioria (tokens de tipografia, cor primária #FF7200, Inter Tight, hairlines, cards). Os desvios:

  • Escala do Início. zoom: 0.78 + centralização vertical fazem o dashboard renderizar em escala diferente do resto.
  • Autenticação fora do sistema. Login/recuperar senha não usam os tokens do app (peso de fonte "medium" custom, painel laranja dividido).
  • Raio de borda dos campos. Convergiu para 6–8px após ajustes, mas convém fixar um único valor no futuro guia.
  • Título "Giraffe" vs "Giraffe CRM". Os <title> alternam entre "Giraffe" e "Giraffe CRM" entre telas (ex.: Agentes e Automações usam só "Giraffe").

11Telas que existem mas ainda não estão maduras

  • Relatórios da empresa — dados sintéticos, contadores conflitantes ("13 pipes"/10; "3769"/27 linhas). Estrutura ok, conteúdo não.
  • Tarefas e Solicitações — só o estado vazio existe; falta o caso preenchido, que é o que dá sentido à tela.
  • Início — depende de zoom, catálogo de pipes divergente, AI Builder não conectado.
  • Painel → Estatísticas / Auditoria / Financeiro — presentes na navegação; conteúdo é ilustrativo e precisa de definição de métricas reais.
  • Notificações (página) — feed estático desalinhado com o popover.

12Dados ou estados duplicados

  • Feed de notificações — duplicado e divergente entre popover (7 itens, datas absolutas) e página (15 itens, tempo relativo). Deveria ser uma fonte única.
  • Cabeçalho/topbar + menu de usuário — copiado literalmente em ~8 telas. Não é erro num protótipo, mas na implementação deve virar um componente único (hoje divergem: ex. nome "Martins junior" vs "Giraffe Marketing").
  • Lista de pipes — repetida em Relatórios, Automações e Perfil, com contagens diferentes.
  • Popover de notificações — o mesmo bloco HTML fixo está embutido em cada página em vez de derivar de dados.

13Regras de negócio que precisam virar documentação

Regras hoje implícitas no comportamento dos protótipos.

  • Movimentação de card entre fases — regras de "mover" (fase inicial/final, cards conectados obrigatórios, preenchimento automático de campos) existem no modal "Configurar mover cards" mas não estão escritas.
  • Estados de card — "atrasado", "expirado", "vencido", "finalizado", "arquivado": definir o que dispara cada um e como se relacionam com vencimento.
  • Automação = Evento → Condição → Ação — catálogo de eventos, condições e ações precisa ser documentado (hoje só existe nos menus).
  • Permissões por papel — "Super Administrador" e o painel de permissões (Automação, Webhooks, Configurações, Acesso total) precisam de matriz papel×capacidade.
  • Formulário inicial e visibilidade — "Público para todos com o link" vs. acessos restritos: regras de publicação e captura.
  • Notificações — quais eventos notificam, por e-mail vs. in-app, padrão vs. específico por pipe (a UI de preferências já pressupõe essas regras).
  • Agentes de IA — o que é "Comportamento", "Conhecimento", quando um agente age, e limites (o "básico" da Fase 1).

14O que falta para a IA interpretar o projeto corretamente

Para que uma IA (ou um novo dev) interprete o Giraffe sem adivinhar, falta a camada de especificação por baixo dos protótipos:

  • Glossário de entidades — Organização, Membro, Papel, Pipe, Fase, Card, Campo, Formulário, Database, Registro, Automação (Evento/Condição/Ação), Agente de IA (Comportamento/Conhecimento), E-mail/Template, Notificação, Tarefa, Solicitação, Conexão, Log/Auditoria.
  • Diagrama de relacionamentos — cardinalidade (um Pipe tem N Fases; uma Fase tem N Cards; um Card pertence a 1 Pipe; Conexões ligam Cards entre Pipes/Databases).
  • Dicionário de dados por entidade — atributos, tipos, obrigatoriedade, valores possíveis (ex.: estados de card).
  • Fonte de dados única e realista — um seed coerente (mesma lista de pipes, usuários, notificações) que todas as telas consumam, em vez de dados inventados por tela.
  • Máquina de estados — de card (fases + atrasado/vencido/arquivado) e de automação (ativo/inativo, execução, log).
  • Matriz de permissões — papel × capacidade, para a IA saber o que cada usuário pode fazer.
Em resumo
Os protótipos respondem "como se parece". Falta o documento que responde "o que é e como se comporta". Esse é o insumo que uma IA precisa para não alucinar regras.

15Próximos passos recomendados

Em ordem de prioridade. Nada aqui exige Fase 2.

Agora (destrava tudo)

  • Escrever o modelo conceitual — glossário + diagrama de entidades + dicionário de dados. Preencher docs/01-documentacao-base/ (as seções 01–06 ausentes).
  • Definir a hierarquia de identidade — Organização → Membro → Papel; decidir se multi-empresa é Fase 1. Corrigir "Martins junior" vs "Giraffe Marketing".
  • Criar um seed de dados único — uma lista canônica de pipes, usuários e notificações consumida por todas as telas.

Em seguida (coerência entre telas)

  • Unificar o catálogo de pipes ("13 pipes" = 13, mesmos nomes em toda parte).
  • Unificar a fonte de notificações (popover e página iguais; badge dinâmico).
  • Resolver a contradição "Tudo em dia" vs. cards atrasados.
  • Gatear os vazamentos de Fase 2 (Token/GraphQL, requisição HTTP) com "Em breve", como já se faz no Painel → API.
  • Documentar as regras de negócio da seção 13.

Depois (polimento)

  • Tornar a busca funcional (ou marcá-la claramente como Fase futura).
  • Datepicker; remover zoom do Início; unificar topbar/menu como componente.
  • Estado preenchido de Tarefas e Solicitações; fluxo de "Sair" → login.
  • Alinhar Login ao sistema de design do app.
